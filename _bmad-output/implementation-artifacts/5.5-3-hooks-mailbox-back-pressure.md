# Story 5.5.3: Hooks Mailbox Back-Pressure

Status: done

## Story

As the harness runtime,
I want event hook dispatch to have back-pressure and timeout protection,
so that rapid event fire from concurrent agents cannot flood the Hooks GenServer mailbox or accumulate unbounded tasks.

## Context

The `Hooks.event/2` function broadcasts via PubSub, and the Hooks GenServer's `handle_info` spawns a `Task.Supervisor.start_child` for each handler — fire-and-forget, no timeout, no limit. With 100 tool calls per agent and multiple concurrent agents, this can spawn hundreds of tasks in rapid succession. If handlers are slow (e.g., knowledge capture involves LLM calls), tasks accumulate and the TaskSupervisor may hit capacity.

Identified in the Epic 5 retrospective as a "Should" action item for Epic 5.5.

## Acceptance Criteria

### AC1: Event Handlers Have a Timeout

**Given** an event hook handler is registered
**When** it is triggered and runs longer than the configured timeout
**Then** the handler task is killed after the timeout
**And** a warning is logged identifying the slow handler

### AC2: Task Spawn Failure Is Handled

**Given** the TaskSupervisor is at capacity or unavailable
**When** an event hook fires and `Task.Supervisor.start_child` fails
**Then** the failure is logged as a warning
**And** the Hooks GenServer continues processing (no crash)

### AC3: Mailbox Depth Is Observable

**Given** the Hooks GenServer is processing events
**When** its mailbox exceeds a configurable threshold
**Then** a warning is logged with the current mailbox size
**And** processing continues (log, don't drop)

### AC4: Event Handler Timeout Is Configurable

**Given** the test environment
**When** tests configure `event_handler_timeout` via app env
**Then** event handler tasks use the configured timeout

### AC5: Test Baseline

**Given** all changes are applied
**When** `mix test` runs
**Then** all existing tests pass with 0 failures, 0 regressions
**And** Credo strict passes with 0 issues

## Dev Notes

### Architecture: Non-Blocking Event Handler Timeout

The current pattern spawns detached tasks (`start_child`) with no monitoring. We need timeouts but can't block `handle_info` waiting for them. Two approaches:

**Option A (recommended): Spawn a monitor task per handler**
```elixir
Task.Supervisor.start_child(Familiar.TaskSupervisor, fn ->
  task = Task.Supervisor.async_nolink(Familiar.TaskSupervisor, fn ->
    handler.fn.(payload)
  end)
  case Task.yield(task, event_handler_timeout()) || Task.shutdown(task, :brutal_kill) do
    {:ok, _} -> :ok
    {:exit, reason} -> Logger.warning("...")
    nil -> Logger.warning("... timed out")
  end
end)
```

Each handler gets a wrapper task that monitors it. The Hooks GenServer returns immediately from `handle_info`. The wrapper task handles timeout/cleanup. This adds one extra task per handler but keeps the GenServer non-blocking.

**Option B: Track tasks in state, check in periodic timer**
More complex, harder to test, and the extra complexity isn't needed for MVP.

### Mailbox Warning Rate Limiting

To avoid flooding logs when the mailbox is large (every message would trigger the warning), track `last_mailbox_warning` timestamp in GenServer state. Only log if more than 10 seconds have passed since the last warning.

### Configurable Values for Test

Set short timeouts in `config/test.exs`:
```elixir
config :familiar, Familiar.Hooks,
  handler_timeout: 50,
  event_handler_timeout: 50,
  mailbox_warning_threshold: 5
```

Note: `handler_timeout: 50` is already set from Story 5.5.1.
