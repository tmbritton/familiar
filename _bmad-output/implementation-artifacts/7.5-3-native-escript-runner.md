# Story 7.5-3: Native Escript Runner

Status: ready-for-dev

## Story

As a developer,
I want to run `fam` as a native CLI tool without Docker,
so that commands execute instantly without container boot overhead and I can run multiple instances across projects.

## Context

The Docker approach has significant UX issues: 5-second cold boot per command, root-owned files, single instance limitation, and no persistent BEAM node without a release. For a developer running Familiar on their own machine against their own code, the safety extension's project directory sandboxing is sufficient — Docker is unnecessary overhead.

The escript (`mix escript.build`) already produces a `fam` binary, but `Main.main/1` doesn't start the OTP application — it calls `System.halt` directly. The fix: add `Application.ensure_all_started(:familiar)` at the top of `main/1`, handle database creation/migration, and install default files on first run.

## Acceptance Criteria

### AC1: `fam` Escript Starts the OTP Application

**Given** a built escript (`mix escript.build`)
**When** the user runs `./fam chat`
**Then** the OTP application starts (Repo, PubSub, ToolRegistry, Hooks, etc.)
**And** the command executes normally
**And** the process exits cleanly

### AC2: First-Run Bootstrap

**Given** no `.familiar/` directory exists
**When** the user runs `./fam` for the first time
**Then** `.familiar/` is created with default roles, skills, workflows, and config.toml
**And** the SQLite database is created and migrated

### AC3: Multiple Instances Work

**Given** two terminal windows in different project directories
**When** each runs `fam chat`
**Then** both work independently with their own `.familiar/` databases

### AC4: Environment Variable Config

**Given** `FAMILIAR_PROVIDER=openai_compatible` and `FAMILIAR_API_KEY=sk-xxx`
**When** the escript runs
**Then** the runtime config picks up the env vars
**And** the correct provider adapter is used

### AC5: Install on PATH

**Given** a built escript
**When** copied to `~/.local/bin/fam` or `/usr/local/bin/fam`
**Then** `fam` works from any directory
**And** `.familiar/` is created relative to the current working directory

### AC6: Persistent BEAM Node (Stretch)

**Given** a running `fam` session
**When** subsequent `fam` commands run in the same project
**Then** they connect to the existing BEAM node via Erlang distribution
**And** commands execute instantly without boot overhead

## Dev Notes

### Key Change in Main.main/1

```elixir
def main(argv) do
  Application.ensure_all_started(:familiar)
  # ... existing logic
end
```

### Bootstrap in Escript

The escript can't run `mix ecto.create` (no Mix). Instead:
- Check if `.familiar/familiar.db` exists
- If not, use `Ecto.Adapters.SQLite3` to create it directly
- Run migrations programmatically via `Ecto.Migrator`

### Persistent Node (AC6)

For instant subsequent commands, the first `fam` invocation could:
1. Start a named BEAM node: `--sname familiar_<project_hash>`
2. Stay alive in the background after the command completes
3. Subsequent `fam` commands detect the running node and connect via `:rpc.call`
4. Auto-shutdown after inactivity timeout

This eliminates boot overhead for all commands after the first.

### Docker Still Available

The Dockerfile remains for CI, untrusted projects, or users who prefer containerization. The escript is the primary distribution method for daily development.
