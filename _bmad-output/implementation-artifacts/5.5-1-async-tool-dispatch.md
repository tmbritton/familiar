# Story 5.5.1: Async Tool Dispatch

Status: done

## Story

As the harness runtime,
I want tool execution to not block the ToolRegistry GenServer,
so that concurrent agents can dispatch tool calls in parallel without serializing through a single process.

## Context

The current `ToolRegistry.dispatch/3` executes the full pipeline — alter hooks, tool function, after-hook event — inside `handle_call`, blocking the GenServer for the entire duration. Every agent's tool call serializes through this single process. With concurrent agents (PM + multiple workers), this becomes a bottleneck: one slow `run_command` blocks all other agents' `read_file` calls.

**This is the critical path item before Phase 3.** Workflows with parallel steps (Epic 6) and multi-agent orchestration require non-blocking dispatch.

## Acceptance Criteria

### AC1: Tool Execution Does Not Block the GenServer

**Given** the ToolRegistry GenServer receives a dispatch call
**When** it resolves the tool function and runs the alter hook pipeline
**Then** it replies to the caller with the result via `GenServer.reply/2` from a spawned Task, not from inside `handle_call`

### AC2: Alter Hooks Still Run Synchronously Before Execution

**Given** a `before_tool_call` alter hook is registered (e.g., safety extension)
**When** a tool is dispatched
**Then** the alter pipeline runs to completion (or veto) before the tool function executes
**And** vetoed calls still return `{:error, {:vetoed, reason}}` immediately

### AC3: After-Hook Events Still Fire on Completion

**Given** a tool call completes (success or error)
**When** the result is returned to the caller
**Then** `after_tool_call` event is broadcast via `Hooks.event/2` as before

### AC4: Tool Crashes Are Isolated

**Given** a tool function raises an exception
**When** it is executing in the spawned Task
**Then** the error is caught and returned as `{:error, {:tool_crashed, message}}`
**And** the ToolRegistry GenServer is unaffected

### AC5: Callers See the Same API

**Given** existing code calls `ToolRegistry.dispatch/3`
**When** the async implementation is active
**Then** the return value contract is unchanged: `{:ok, result}` | `{:error, reason}`
**And** callers block on their `GenServer.call` until the result is ready (no API change)

### AC6: Concurrent Dispatch Works

**Given** two agents dispatch tool calls simultaneously
**When** both calls reach the ToolRegistry
**Then** both tool functions execute concurrently (not serialized)
**And** each caller receives its own correct result

### AC7: Test Baseline

**Given** all changes are applied
**When** `mix test` and `mix test --include integration` run
**Then** all existing tests pass with 0 failures, 0 regressions
**And** Credo strict passes with 0 issues

## Dev Notes

### Architecture: The `GenServer.reply/2` Pattern

The key insight is that `GenServer.call` blocks the *caller*, not just the GenServer. When we use `GenServer.reply/2` from a Task, the caller still blocks waiting for the reply — but the GenServer is free to handle other calls. This gives us concurrency without changing the caller's API.

```
Before (blocking):
  AgentA → GenServer.call(:dispatch) → [handle_call blocks doing work] → reply
  AgentB → GenServer.call(:dispatch) → [waiting for GenServer] → ...

After (async):
  AgentA → GenServer.call(:dispatch) → handle_call spawns Task, returns :noreply
                                        Task does work → GenServer.reply(from, result)
  AgentB → GenServer.call(:dispatch) → handle_call spawns Task, returns :noreply
                                        Task does work → GenServer.reply(from, result)
  Both Tasks run concurrently.
```

### What Stays Synchronous Inside handle_call

1. **Tool lookup** (`fetch_tool/2`) — map lookup, nanoseconds
2. **Alter hook pipeline** (`run_before_hook/3`) — calls `Hooks.alter/3` which is itself a `GenServer.call` to the Hooks GenServer. The Hooks module already isolates each handler in a `Task.Supervisor.async_nolink` with 5s timeout and circuit breaker. This is fast (typically <10ms for safety checks). Keeping it synchronous in the dispatch path ensures vetoes prevent tool execution.

### What Moves to the Spawned Task

1. **Tool function execution** (`tool.function.(args, context)`) — this is the slow part (file I/O, shell commands, LLM calls for `spawn_agent`)
2. **After-hook broadcast** (`Hooks.event(:after_tool_call, ...)`) — fire-and-forget, but logically belongs after execution

### Critical Implementation Detail

The `from` parameter from `handle_call` must be captured and passed to the Task closure. The Task calls `GenServer.reply(from, result)` — this is a well-documented OTP pattern. The GenServer never processes the Task's completion (no `handle_info` needed for this).

### Existing Patterns to Reuse

- `Task.Supervisor.async_nolink(Familiar.TaskSupervisor, fn -> ... end)` — used in AgentProcess for LLM calls, Hooks for alter handler isolation
- try/rescue for crash isolation — used in current `execute_tool/3`
- `broadcast_after_hook/3` already has rescue isolation

### Per-Test Isolation

The `tool_registry_test.exs` already uses per-test named GenServers with `start_supervised!`. The `dispatch_tool/5` helper bypasses the global GenServer. New concurrent tests should use the same pattern. For the global-dispatch tests (`after_tool_call event` describe block), use unique tool names as currently done.
