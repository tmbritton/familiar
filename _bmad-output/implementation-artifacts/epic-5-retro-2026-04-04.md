## Successes

1. **Single generic AgentProcess proved the thesis.** One GenServer module runs analyst, coder, reviewer — all differentiation via markdown role files. No per-role Elixir code. This is the core reason BEAM was chosen and it works.

2. **Extension system delivered real separation of concerns.** Safety enforcement and knowledge capture are extensions, not hardcoded. The alter/event hook split (synchronous veto vs async observation) proved elegant — safety vetoes tool calls without knowledge capture needing to know or care.

3. **Strict write sequence for crash safety.** Log intent → stat check → write → log completion. The invariant is simple, testable, and crash-safe via SQLite WAL. Git-based rollback for tracked files was Buddy's idea and avoids storing file content in the database.

4. **Story-to-story intelligence transfer.** Every story's Dev Notes section grew richer, citing patterns from earlier stories. By Story 5.8, the dev agent could reuse `require_arg/2`, `get_arg/2`, role-based LLM mocks, per-test DynamicSupervisor, and `:counters` for call sequencing — all discovered and documented in earlier stories.

5. **Code review caught real bugs.** signal_ready registration never called (5.8), undo_write dead branch (5.9), AgentProcess missing task_id (5.10), format_previous_steps reverse order (5.11). The 3-layer parallel review (Blind Hunter, Edge Case Hunter, Acceptance Auditor) consistently found issues the dev agent missed.

6. **Property-based testing for invariants.** StreamData proved rollback leaves filesystem consistent (5.9), output never exceeds token budget (5.4), and content hashing is deterministic (5.9). These properties catch edge cases unit tests miss.

7. **Per-test DynamicSupervisor pattern.** Buddy's question — "Can each test have its own DynamicSupervisor?" — solved the flaky test problem in Story 5.8 and became the standard for all subsequent integration tests.

8. **Party mode for architectural decisions.** The async tool dispatch discussion (pre-5.8.5) used party mode to get perspectives from Winston (architect), Amelia (dev), Murat (QA), and Bob (SM). Consensus: build builtins now, defer GenServer.reply/2 to Epic 5.5. This was the right call.

## Challenges

1. **Agent restart consuming mock expectations.** Story 5.8 — `AgentProcess` used default `restart: :permanent`. Dead agents restarted by DynamicSupervisor, consumed Mox expectations meant for the next step. Root cause took significant debugging. Fix: `use GenServer, restart: :temporary` — agents are one-shot tasks. This should have been the default from the start.

2. **ETS concurrency is deceptively hard.** Three separate TOCTOU races discovered across Safety init, WorkflowRunner registry, and extension loading. Each individually unlikely, collectively a pattern. The check-then-act anti-pattern with ETS showed up in every module that used named tables. Story 5.11 replaced all instances with atomic try/catch.

3. **Silent error accumulation.** `Conversations.add_message` return values ignored in 4 call sites. Extension loader silently dropped unrecognized hook types. Story 5.11's `log_add_message` wrapper immediately caught a real bug — nil content assistant messages were failing to persist, completely invisible until we added logging.

4. **O(n²) list append is intuitive but wrong.** `state.messages ++ [msg]` appeared in 3 places across AgentProcess and WorkflowRunner. Each individually acceptable for MVP limits (100 tool calls, <20 steps), but it's the wrong pattern. Story 5.11 converted all to prepend+reverse.

5. **AgentProcess doesn't naturally provide task_id.** Discovered in Story 5.10 integration testing — file writes always bypassed the transaction module because `context.task_id` was nil. The agent_id-as-task_id solution works for now but a proper task management system (Epic 4) would provide real task IDs.

## Key Insights

1. **The alter/event hook split is powerful.** Alter hooks (synchronous, can veto) enforce safety. Event hooks (async, fire-and-forget) capture knowledge. This separation lets safety be strict without blocking observational concerns. The priority system means safety always runs first.

2. **Markdown-defined agents work.** Role files define system prompts, skill files define tool-specific instructions. The Elixir code never knows what an "analyst" or "coder" does — it just assembles prompts, runs the LLM, and dispatches tool calls. This is the architecture's key flexibility.

3. **Integration tests catch architectural gaps.** Story 5.10's golden path test revealed that file transactions don't flow end-to-end through the agent because task_id wasn't injected. Unit tests on individual components couldn't catch this — only the integration test exposed the seam.

4. **Code review is highest-value QA activity.** 3-layer parallel review found ~45 patches across 8 stories. The most impactful finds: signal_ready never wired (5.8), rollback can't restore originals (5.9 — led to git-based rollback), format_previous_steps wrong order (5.11). Each would have been a production bug.

5. **Mid-sprint story additions are fine.** Story 5.8.5 (builtin tools) emerged when we realized stubs weren't implementations. Story 5.11 (hardening) emerged from accumulated deferred items. Both were necessary and didn't disrupt flow.

## Patterns Established

These patterns are now validated and should be reused in Phase 3:

1. **Behaviour ports for I/O** — FileSystem, Shell, Clock, LLM, Embedder
2. **Extension behaviour** — name/tools/hooks/init callbacks
3. **Alter pipeline for enforcement** — priority-ordered, timeout-isolated, circuit-breaker
4. **Event dispatch for observation** — PubSub, fire-and-forget, crash-isolated
5. **Per-test DynamicSupervisor** — agent isolation in tests
6. **Role-based LLM mock stubs** — match on system prompt content
7. **Strict invariant sequences** — log intent → act → log completion
8. **Property tests for invariants** — budget, rollback consistency, hashing
9. **Prepend+reverse for list accumulation** — O(n) instead of O(n²)
10. **Atomic ETS init** — try/catch on :ets.new, defensive lookups with defaults
