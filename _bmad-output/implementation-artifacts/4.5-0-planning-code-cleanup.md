# Story 4.5.0: Planning Code Cleanup

Status: done

## Story

As a developer building the agent harness,
I want to remove the bespoke planning orchestration code that will be replaced by the workflow runner,
So that the codebase is clean before building the role/skill system and there is no dead code to confuse future implementation.

## Context

The architecture addendum (2026-04-03) reframed planning as a workflow on the agent harness. The survival analysis identified ~700 lines of infrastructure that carries forward and ~550 lines of orchestration/prompts that are replaced. This story removes the code that will be replaced, extracts reusable pieces into appropriate locations, and ensures all tests still pass.

**What stays (infrastructure):**
- `verification.ex` (216 lines) ‚Äî 100% survives, pure functional
- `trail.ex` (125 lines) ‚Äî 100% survives, PubSub infrastructure
- `trail_formatter.ex` (110 lines) ‚Äî 100% survives, pure formatter
- `spec_review.ex` (259 lines) ‚Äî ~95% survives, approval logic
- `session.ex` (37 lines) ‚Äî Ecto schema, reusable
- `message.ex` (47 lines) ‚Äî Ecto schema, reusable
- `spec.ex` (50 lines) ‚Äî Ecto schema, reusable
- `planning.ex` (48 lines) ‚Äî facade, will be thinned

**What goes (orchestration being replaced by workflow runner):**
- `engine.ex` (273 lines) ‚Äî orchestration loop replaced by workflow runner
- `prompt_assembly.ex` (97 lines) ‚Äî prompt content moves to role files; assembly logic rebuilt in Epic 5
- `librarian.ex` (237 lines) ‚Äî GenServer replaced by AgentProcess + role file
- `spec_generator.ex` (465 lines) ‚Äî tool loop replaced by AgentProcess; tool handlers extracted

**What gets extracted before deletion:**
- `spec_generator.ex` tool dispatch implementations (`file_read`, `knowledge_search`) ‚Üí new utility module for reuse as tool handlers in Epic 5
- `spec_generator.ex` frontmatter/title/slug helpers ‚Üí remain in planning context as spec utilities
- `prompt_assembly.ex` `normalize_history/1` ‚Üí utility function preserved in planning context
- `librarian.ex` `merge_results/2`, `format_results_for_llm/1` ‚Üí utility functions for future librarian role

## Dev Notes

### Architecture Constraints

- **Do NOT delete schemas**: `session.ex`, `message.ex`, `spec.ex` are Ecto schemas tied to migrations. They stay. The data model is valid regardless of orchestration approach.
- **Do NOT delete surviving infrastructure**: `verification.ex`, `trail.ex`, `trail_formatter.ex`, `spec_review.ex` all survive intact per the addendum survival analysis.
- **Do NOT delete migrations**: The `planning_sessions`, `planning_messages`, `planning_specs` tables remain.
- **Do NOT add `Familiar.AgentSupervisor`**: That's Story 4.5-3's job. Just remove `LibrarianSupervisor` ‚Äî Epic 5 adds the replacement.
- **Stub, don't break**: CLI commands that are removed should return a clear "not yet available" message, not crash. Users who have been using `fam plan` should get a helpful message.

### What This Unblocks

- Story 4.5-1 (Roles Context) can build in a clean codebase without dead orchestration code
- Story 4.5-3 (Inline Prompt Migration) has nothing to migrate ‚Äî the inline prompts are already gone
- Epic 5 builds the AgentProcess and workflow runner on a clean foundation

### Impact on Story 4.5-3

With this cleanup, Story 4.5-3 scope reduces significantly:
- No `@planning_system_prompt` to migrate (already deleted)
- No librarian inline prompts to migrate (already deleted)
- 4.5-3 becomes: rename `LibrarianSupervisor` ‚Üí `AgentSupervisor` (if not done here) and wire `PromptAssembly` to accept role (if a new PromptAssembly is needed)
- Consider merging 4.5-3 into this story or 4.5-2

### Key Patterns for Epic 5 Reference (Document in Completion Notes)

The following patterns from deleted code should be captured for the Epic 5 developer:

1. **Tool dispatch loop** (`spec_generator.ex:do_tool_loop/5`): send prompt ‚Üí receive tool calls ‚Üí dispatch each ‚Üí append results ‚Üí repeat. Max rounds enforced.
2. **file_read tool handler**: validate path (no traversal), read via FileSystem behaviour, filter via SecretFilter, return content or error.
3. **knowledge_search tool handler**: call Knowledge.search/2, format results as text with source citations.
4. **Trail event broadcasting**: `call_trail/3` pattern with rescue for fire-and-forget.
5. **Message persistence**: best-effort via `Repo.insert`, failures logged not raised.
