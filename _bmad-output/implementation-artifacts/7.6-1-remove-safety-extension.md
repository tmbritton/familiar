# Story 7.6-1: Remove Safety Extension

Status: done

## Story

As a Familiar maintainer,
I want the `Familiar.Extensions.Safety` module and every integration point deleted,
So that the codebase stops providing a false-confidence safety layer that an adversarial LLM can trivially bypass.

## Acceptance Criteria

1. **AC1: Safety module deleted.** `familiar/lib/familiar/extensions/safety.ex` does not exist. `familiar/test/familiar/extensions/safety_test.exs` does not exist.

2. **AC2: Config updated.** `config/config.exs` default extensions list contains only `[Familiar.Extensions.KnowledgeStore]` — Safety is removed.

3. **AC3: Boundary module updated.** `familiar/lib/familiar/extensions/extensions.ex` no longer exports `Familiar.Extensions.Safety`.

4. **AC4: Source references cleaned.** No file under `familiar/lib/` contains the atom `Familiar.Extensions.Safety` or alias `Extensions.Safety`. Comments mentioning the Safety extension by module name are removed or rewritten. Generic safety *comments* (e.g., "safety margin" in `paths.ex`) are untouched.

5. **AC5: Default role files updated.** The `## Safety` section in default role prompts (embedded in `familiar/lib/familiar/knowledge/default_files.ex`) is either removed or reworded to reflect the new sandboxing posture — no references to a runtime Safety extension enforcing constraints. Acceptable replacement: a note that the user is responsible for sandboxing and Familiar has no runtime safety layer.

6. **AC6: Tests updated.** All test files that referenced `Familiar.Extensions.Safety` compile and pass without it:
   - `harness_integration_test.exs` — load only `KnowledgeStore`; remove the "safety veto" describe block entirely.
   - `path_resolution_defaults_test.exs` — remove the `Familiar.Extensions.Safety default project_dir` describe block.
   - `default_files_test.exs` — update the "coder prompt contains safety rules" test to match the new wording (or remove if the Safety section was deleted from the prompt).

7. **AC7: Hook/registry infrastructure untouched.** The `Familiar.Extension` behaviour, `Hooks.register_alter_hook/4`, `Hooks.alter/3`, the `:before_tool_call` / `:after_tool_call` hook points, and the `{:halt, reason}` return contract remain fully functional. Tests in `hooks_test.exs`, `tool_registry_test.exs`, and `agent_process_test.exs` that use `"safety"` or `"test-safety"` as label strings continue to pass unchanged — these test the hook *infrastructure*, not the Safety *extension*.

8. **AC8: ETS table gone.** No code creates or references `:familiar_safety_config`.

9. **AC9: Clean toolchain.** `mix compile --warnings-as-errors`, `mix format --check-formatted`, `mix credo --strict`, `mix test`, `mix dialyzer` all pass with zero warnings/failures.

10. **AC10: Stress-tested.** Every touched test file passes 50 consecutive runs with no flakes.

### What NOT to touch

- **Hook infrastructure** (`hooks.ex`, `tool_registry.ex` core logic) — the `before_tool_call` alter pipeline, `{:halt, reason}` return, priority ordering, timeout/circuit-breaker all stay. Only comments/docs referencing Safety-as-extension change.
- **Test label strings** — `hooks_test.exs`, `tool_registry_test.exs`, `agent_process_test.exs` use `"safety"` and `"test-safety"` as arbitrary string labels in test data. These test the hook *infrastructure*, not the Safety *extension*. Leave them unchanged.
- **Secret filtering** (`lib/familiar/knowledge/secret_filter.ex`, Story 2-6) — this is data hygiene, not execution safety. Stays.
- **KnowledgeStore extension** — completely independent of Safety. Only change: it becomes the sole default extension.
- **Planning artifacts** — Story 7.6-3 handles doc updates separately.

### Architecture context

The Safety extension is an `alter` hook at priority 1 in the `before_tool_call` pipeline. Removing it means the alter pipeline still runs but has no built-in handlers (only KnowledgeStore's event hooks remain). The `Familiar.Extension` behaviour defines: `name/0`, `tools/0`, `hooks/0`, `init/1`, `child_spec/1`. KnowledgeStore satisfies all callbacks. The extension loader in `application.ex` line 147 calls `load_extensions()` which reads from `Application.get_env(:familiar, :extensions, [])`.

Safety used an ETS table (`:familiar_safety_config`) for thread-safe config reads. No other module reads this table. Deleting Safety removes the ETS table creation; no cleanup needed.
