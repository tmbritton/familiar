Status: done

## Story

As a user,
I want the system to install well-crafted default role and skill files during project initialization,
So that agents work out of the box while I can customize them later.

## Acceptance Criteria

### AC1: Default Role Files Installed

**Given** `fam init` runs on a new project
**When** `.familiar/` directory is created via `DefaultFiles.install/1`
**Then** `.familiar/roles/` contains 6 MVP role files with proper YAML frontmatter:
  - `analyst.md` — interactive planning conversation agent
  - `coder.md` — autonomous code implementation agent
  - `reviewer.md` — autonomous code review and knowledge extraction agent
  - `librarian.md` — multi-hop knowledge retrieval and summarization agent
  - `archivist.md` — post-task knowledge capture agent
  - `project-manager.md` — batch orchestration, worker coordination, progress summarization agent
**And** each role file body contains a full system prompt (not stubs — detailed instructions for the agent's behavior)

### AC2: Default Skill Files Installed

**Given** `fam init` runs on a new project
**When** `.familiar/` directory is created via `DefaultFiles.install/1`
**Then** `.familiar/skills/` contains MVP skill files with proper YAML frontmatter and tool references:
  - `implement.md`, `test.md`, `research.md` — coder skills
  - `review-code.md` — reviewer skill
  - `search-knowledge.md`, `summarize-results.md` — librarian skills
  - `extract-knowledge.md`, `capture-gotchas.md` — archivist skills
  - `dispatch-tasks.md`, `monitor-workers.md`, `summarize-progress.md`, `evaluate-failures.md` — project-manager skills
**And** each skill file has `name`, `description`, `tools` in frontmatter and instruction text in the body
**And** skill `tools` lists reference tools from the MVP tool set where possible (forward-declared tools like `spawn_agent`, `monitor_agents`, `broadcast_status`, `signal_ready` produce warnings, not errors)

### AC3: Re-init Safety (Existing Behavior Preserved)

**Given** default files already exist (re-init or upgrade)
**When** `DefaultFiles.install/1` runs
**Then** existing role/skill files are NOT overwritten (user edits preserved)
**And** new files that don't yet exist ARE created (additive install)

### AC4: Installed Files Pass Validation

**Given** default files are installed
**When** `Roles.validate_role/2` is called on each installed role
**Then** all roles pass validation (all referenced skills exist as files)

**Given** default files are installed
**When** `Roles.validate_skill/2` is called on each installed skill
**Then** all skills pass validation (unknown tools produce warnings, not errors)

### AC5: Full System Prompts (Not Stubs)

**Given** the librarian role file is installed
**Then** its system prompt contains search refinement and summarization instructions (the knowledge the deleted `librarian.ex` inline prompts contained)

**Given** the analyst role file is installed
**Then** its system prompt contains planning conversation instructions (the knowledge the deleted `prompt_assembly.ex` inline prompts contained)

**Given** the coder role file is installed
**Then** its system prompt contains implementation guidance with safety rules and convention adherence

### AC6: Test Coverage

**Given** the updated DefaultFiles module is implemented
**When** `mix test` runs
**Then** all installed role files are verified to exist after install
**And** all installed skill files are verified to exist after install
**And** all role files pass `Roles.load_role/2` (valid YAML frontmatter, required fields)
**And** all skill files pass `Roles.load_skill/2` (valid YAML frontmatter, required fields)
**And** cross-validation passes: every role's skill references resolve to installed skill files
**And** the non-overwrite behavior is preserved for both roles and skills

## Dev Notes

### Architecture Constraints

- **Role file format is locked** — frontmatter fields: `name` (required), `description` (required), `skills` (required), `model` (optional, default: "default"), `lifecycle` (optional, default: "ephemeral"). Body is system prompt. See `Familiar.Roles.Loader.build_role/1` for parsing.
- **Skill file format is locked** — frontmatter fields: `name` (required), `description` (required), `tools` (required), `constraints` (optional, default: {}). Body is instructions. See `Familiar.Roles.Loader.build_skill/1` for parsing.
- **Skill names in role frontmatter must match skill filenames** (without `.md`). E.g., `skills: [search-knowledge]` → `.familiar/skills/search-knowledge.md` must exist.
- **Tool names in skill frontmatter are validated against `@mvp_tools`** in `validator.ex`: `read_file`, `write_file`, `list_files`, `search_files`, `run_shell`, `search_context`, `store_context`. Tools not in this list produce `Logger.warning` but skill loads successfully (forward-declaration).
- **`DefaultFiles.install/1` uses non-overwrite semantics** — `File.exists?` check before each write. This is the existing behavior; do not change it.

### Key Design Decisions

1. **Skill filenames use hyphens, not underscores** — e.g., `search-knowledge.md` not `search_knowledge.md`. The `name` field in frontmatter uses the same convention. This matches the role filename convention (`project-manager.md`).

2. **Forward-declared tools are expected** — Skills for project-manager (e.g., `dispatch-tasks.md`) reference `spawn_agent`, `monitor_agents`, `broadcast_status`, `signal_ready` which are not in the current `@mvp_tools`. These are Epic 5 tools. `validate_skill/2` will produce Logger.warning for each — this is correct behavior, not a bug.

3. **No workflow file changes needed** — The 3 existing workflow files stay as-is. This story only adds roles and skills.

4. **System prompt quality matters** — These prompts will drive actual agent behavior in Epic 5+. Write them as real instructions an LLM would follow, not documentation placeholders. Reference the architecture Decision A2 examples for tone and detail level.

5. **Module attribute string heredocs** — Each role/skill is a string in an Elixir map attribute. Use `"""..."""` heredocs with proper indentation. The content must have `---` delimiters at the start of lines (not indented by Elixir's heredoc leading whitespace trimming).
