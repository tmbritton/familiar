# Story 7.5-1: OpenAI-Compatible Provider Adapter

Status: done

## Story

As a developer who doesn't have local LLM hardware,
I want Familiar to work with remote OpenAI-compatible APIs (DeepSeek, Qwen, etc.),
so that I can manually test the agent harness against real LLMs over HTTP.

## Context

Familiar's provider system is already abstracted via the `Familiar.Providers.LLM` behaviour. The current implementation is `OllamaAdapter` which targets a local Ollama instance. DeepSeek and Qwen both expose OpenAI-compatible `/v1/chat/completions` endpoints with tool calling support. A single adapter handles both.

For initial manual testing, only `chat/2` is needed. Streaming and embeddings can be stubbed. The knowledge store (which needs embeddings) is not required for `fam chat`, `fam plan/do/fix`, or any CLI management commands.

## Acceptance Criteria

### AC1: OpenAI-Compatible Chat Works

**Given** a valid API key and base URL for DeepSeek or Qwen
**When** `Familiar.Providers.chat/2` is called
**Then** messages are sent to `/v1/chat/completions` with the correct format
**And** the response is parsed into `%{content: String.t(), tool_calls: [map()]}`
**And** errors return `{:error, {:provider_unavailable, map()}}`

### AC2: Tool Calling Works

**Given** the agent prompt includes tool definitions
**When** the LLM returns a tool call in OpenAI format
**Then** the tool call is parsed into `%{"name" => name, "arguments" => args}`
**And** AgentProcess can dispatch it through ToolRegistry

### AC3: Config via Environment Variables

**Given** environment variables `FAMILIAR_API_KEY`, `FAMILIAR_BASE_URL`, `FAMILIAR_CHAT_MODEL`
**When** the adapter is configured
**Then** it reads from env vars with sensible defaults
**And** the config can be overridden per-call via opts

### AC4: Stub Embedder for Testing Without Embeddings

**Given** no embedding provider is available
**When** the stub embedder is configured
**Then** it returns zero vectors of the correct dimension
**And** knowledge store operations degrade gracefully (no semantic search, but CRUD works)

### AC5: Switchable via Config

**Given** the new adapter
**When** `config :familiar, Familiar.Providers.LLM, Familiar.Providers.OpenAICompatibleAdapter` is set
**Then** all agent operations use the remote provider
**And** switching back to Ollama is a config-only change

### AC6: Test Baseline

**Given** all changes are applied
**When** `mix test` runs
**Then** all existing tests pass (tests use LLMMock, unaffected by adapter choice)
**And** Credo strict and Dialyzer pass with 0 issues

## Dev Notes

### OpenAI Chat Completions Request Format

```json
{
  "model": "deepseek-chat",
  "messages": [
    {"role": "system", "content": "You are a helpful assistant."},
    {"role": "user", "content": "Hello"}
  ],
  "tools": [
    {
      "type": "function",
      "function": {
        "name": "read_file",
        "description": "Read a file",
        "parameters": {"type": "object", "properties": {"path": {"type": "string"}}}
      }
    }
  ]
}
```

### OpenAI Chat Completions Response Format

```json
{
  "choices": [{
    "message": {
      "role": "assistant",
      "content": "Hello! How can I help?",
      "tool_calls": [{
        "id": "call_abc123",
        "type": "function",
        "function": {
          "name": "read_file",
          "arguments": "{\"path\": \"src/main.ex\"}"
        }
      }]
    }
  }]
}
```

### Familiar's Internal Format

AgentProcess expects from the provider:
```elixir
{:ok, %{
  content: "Hello! How can I help?",
  tool_calls: [
    %{"function" => %{"name" => "read_file", "arguments" => %{"path" => "src/main.ex"}}}
  ]
}}
```

The tool_calls format is already OpenAI-compatible — AgentProcess's `extract_tool_call/1` handles both string and atom keys.

### Provider Differences

| Setting | DeepSeek | Qwen |
|---------|----------|------|
| Base URL | `https://api.deepseek.com` | `https://dashscope.aliyuncs.com/compatible-mode` |
| Chat model | `deepseek-chat` | `qwen-plus` |
| API key env | `DEEPSEEK_API_KEY` | `DASHSCOPE_API_KEY` |

Use generic env vars (`FAMILIAR_API_KEY`, etc.) so the same config works for any OpenAI-compatible provider.
