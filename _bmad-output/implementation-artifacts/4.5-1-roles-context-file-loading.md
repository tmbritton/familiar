# Story 4.5.1: Roles Context & File Loading

Status: ready-for-dev

## Story

As a developer building the agent harness,
I want a `Familiar.Roles` context that loads and validates role and skill markdown files from `.familiar/roles/` and `.familiar/skills/`,
So that the AgentProcess (Epic 5) can load any agent's behavior from user-editable files at runtime.

## Acceptance Criteria

### AC1: Role File Loading

**Given** a valid role markdown file exists in `.familiar/roles/`
**When** `Roles.load_role("coder", familiar_dir: path)` is called
**Then** YAML frontmatter is parsed for: `name`, `description`, `model`, `lifecycle`, `skills`
**And** the markdown body below the closing `---` is returned as `system_prompt`
**And** `{:ok, %Role{name, description, model, lifecycle, skills, system_prompt}}` is returned
**And** `model` defaults to `"default"` when absent
**And** `lifecycle` defaults to `:ephemeral` when absent

**Given** a role file does not exist
**When** `Roles.load_role("nonexistent", familiar_dir: path)` is called
**Then** `{:error, {:role_not_found, %{name: "nonexistent"}}}` is returned

### AC2: Skill File Loading

**Given** a valid skill markdown file exists in `.familiar/skills/`
**When** `Roles.load_skill("implement", familiar_dir: path)` is called
**Then** YAML frontmatter is parsed for: `name`, `description`, `tools`, `constraints`
**And** the markdown body below the closing `---` is returned as `instructions`
**And** `{:ok, %Skill{name, description, tools, constraints, instructions}}` is returned
**And** `constraints` defaults to `%{}` when absent

### AC3: Listing

**Given** multiple role files exist in `.familiar/roles/`
**When** `Roles.list_roles(familiar_dir: path)` is called
**Then** `{:ok, [%Role{}, ...]}` is returned with all valid roles
**And** invalid files are excluded from the list (logged as warnings, not errors)

**Given** multiple skill files exist in `.familiar/skills/`
**When** `Roles.list_skills(familiar_dir: path)` is called
**Then** `{:ok, [%Skill{}, ...]}` is returned with all valid skills

### AC4: Validation

**Given** a role file with malformed YAML frontmatter
**When** `Roles.load_role(name, familiar_dir: path)` is called
**Then** `{:error, {:invalid_role, %{name: name, reason: "..."}}}` is returned
**And** the error message identifies the specific problem (e.g., "missing required field 'skills'")

**Given** a role references a skill that does not exist in `.familiar/skills/`
**When** `Roles.validate_role(name, familiar_dir: path)` is called
**Then** `{:error, {:invalid_role, %{name: name, reason: "references skill 'foo' which does not exist in .familiar/skills/"}}}` is returned

**Given** a skill references a tool not in the known tool list
**When** `Roles.validate_skill(name, familiar_dir: path)` is called
**Then** a warning is logged but the skill loads successfully (allows forward-declaration of tools not yet registered)

### AC5: Test Coverage

**Given** the Roles context is implemented
**When** `mix test` runs
**Then** all role loading, skill loading, listing, and validation scenarios are covered
**And** YAML frontmatter parsing for all metadata fields is tested
**And** error cases: missing file, malformed frontmatter, missing required fields, bad skill references, bad tool references
**And** near-100% coverage on `Familiar.Roles` context

## Dev Notes

### Architecture Constraints

- **Hexagonal architecture**: This module reads files directly (not through `FileSystem` behaviour) because role/skill files are configuration, not project files. The `FileSystem` behaviour is for agent-operated file I/O within the user's project. Configuration loading uses `File.read/1` directly — same pattern as `Familiar.Config` which reads TOML via `Toml.decode_file/1`.
- **Error tuple convention**: All public functions return `{:ok, result}` or `{:error, {atom_type, map_details}}` — follow exactly.
- **`use Boundary`**: Required on the `Familiar.Roles` facade module. `deps: []` (no other contexts needed), `exports: [Familiar.Roles]`.
- **No Ecto/DB**: This context is purely file-based. No schemas, no migrations, no Repo calls.
- **Logger for warnings**: Unknown tool references log via `Logger.warning/1`, not `IO.puts`. Follow existing codebase convention.

### Existing Code to Reuse

| What | Where | How to Reuse |
|------|-------|-------------|
| YAML parsing library | `{:yaml_elixir, "~> 2.9"}` in mix.exs | Already a dependency — `YamlElixir.read_from_string/1` |
| Frontmatter split pattern | `spec_review.ex:replace_frontmatter_status/2` | Same `---` delimiter splitting logic — `String.split(content, "\n")`, find `"---"` lines |
| Config file reading pattern | `config.ex:read_toml/1` | Similar "read file, parse, validate, return struct" flow |
| File-based test pattern | `test/familiar/knowledge/default_files_test.exs` | `@moduletag :tmp_dir` + write fixture files to tmp + test against them |
| Public API facade pattern | `knowledge/knowledge.ex` | `use Boundary`, function specs, delegations to internal modules |

### Key Design Decisions

1. **`familiar_dir` as an opt, not application env**: Pass the `.familiar/` path through opts so tests can point at `tmp_dir`. Default resolves from `File.cwd!()` + `".familiar"`. Same DI pattern used across the codebase (e.g., `file_system` opt, `providers_mod` opt).

2. **Direct file reads, not FileSystem behaviour**: Role/skill files are system configuration. Using the `FileSystem` behaviour (designed for sandboxed agent I/O) would conflate two distinct concerns. `Familiar.Config` sets the precedent: config reads bypass the behaviour.

3. **Known tool list as a default, not a registry call**: Story 4.5-1 hardcodes the MVP tool list as a module attribute for validation. Story 5.1 (ToolRegistry) will provide a runtime query. The validator accepts a `known_tools:` opt for future override.

4. **Lifecycle as atom**: The `lifecycle` string from YAML (`"ephemeral"`, `"batch"`, `"session"`) is converted to an atom. Unknown values produce a validation error, not an atom leak. Use explicit pattern matching, not `String.to_atom/1`.

5. **Body trimming**: The system_prompt/instructions body should be trimmed of leading/trailing whitespace but otherwise preserved as-is (including internal markdown formatting).

6. **List functions filter invalid files**: `list_roles/1` and `list_skills/1` log warnings for invalid files but do not include them in the returned list. The system remains functional even with one bad file.

### Previous Story Intelligence

- **Test baseline**: 733 tests + 14 properties, 0 failures (after Story 3.4)
- **Credo**: strict mode, 0 issues
- **Deferred items from Epic 3 reviews**: `normalize_status` uses unbounded `String.to_atom` — explicitly avoid this pattern. Use `Map.fetch!` with an allowed-values map or explicit case/cond for lifecycle conversion.
- **Epic 3 frontmatter pattern**: `spec_review.ex` already splits markdown files on `---` delimiters and manipulates frontmatter. Reuse the same approach but don't copy the code — the Roles loader is a clean implementation, not a refactor of SpecReview.

### Testing Standards

- `@moduletag :tmp_dir` for file-based tests — write fixture `.md` files in `setup`, call Roles API with `familiar_dir: tmp_dir`
- No Mox needed — this context does direct file I/O (configuration, not project files)
- No DataCase needed — no database access
- `use ExUnit.Case, async: true` should work since tests use isolated tmp directories
- Test each error path explicitly: missing file, bad YAML, missing required field, missing skill reference, unknown tool (warning)
- Test `list_*` with a mix of valid and invalid files — verify invalid are excluded and warnings logged (capture_log)
