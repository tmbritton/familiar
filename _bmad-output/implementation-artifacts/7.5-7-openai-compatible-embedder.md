# Story 7.5-7: OpenAI-Compatible Embedder + Model-Change Reindex

Status: ready-for-dev

## Story

As a Familiar user who configured an OpenRouter (or DeepInfra, or any OpenAI-compatible) provider,
I want a real embedding adapter that produces meaningful vectors and a way to re-embed everything when I switch models,
so that semantic search actually works without forcing me to run a local Ollama instance.

## Context

Today, when `provider.type == "openai_compatible"` Familiar wires the embedder to `Familiar.Providers.StubEmbedder` (`config/runtime.exs:58`), which returns zero vectors. Knowledge store CRUD works, but semantic search returns nothing. The user explicitly flagged this in Epic 7.5 testing — once they switched to OpenRouter, semantic search silently became useless.

The fix is a new `Familiar.Providers.OpenAIEmbedder` adapter that calls the OpenAI-compatible `/embeddings` endpoint (the same shape OpenAI, OpenRouter, DeepInfra, Together AI, and Groq all expose). The recommended default is `openai/text-embedding-3-small` via OpenRouter — $0.02/M tokens, 1536 dimensions, excellent for code and prose retrieval. Reference: `.claude/.../memory/project_embedding_reindex.md`.

**Two complications make this more than a one-file adapter:**

1. **Dimension change.** The current sqlite-vec virtual table is hard-coded to `float[768]` for `nomic-embed-text` (`priv/repo/migrations/20260401200000_create_knowledge_entries.exs:22`) and `Familiar.Knowledge` enforces `@expected_embedding_dimensions 768` (`lib/familiar/knowledge/knowledge.ex:516`). `text-embedding-3-small` is 1536. Mixing dimensions silently corrupts the index, so we have to recreate the virtual table at the new dimension.

2. **Model-change requires reindex.** Different embedding models produce incompatible vector spaces — switching from `nomic-embed-text` to `text-embedding-3-small` makes every existing vector meaningless even if dimensions matched. The system has to (a) record which model produced the current vectors, (b) detect mismatches, and (c) provide a way to re-embed everything.

The story scope is the adapter, the dimension migration, the model-change detection metadata, and a `fam context --reindex` CLI command. Per-call dimension auto-discovery and on-the-fly schema migration are out of scope.

**Why now:** unblocks real semantic search for OpenRouter users. Required before recommending Familiar to anyone who doesn't run Ollama locally.

## Acceptance Criteria

### AC1: `Familiar.Providers.OpenAIEmbedder` implements the Embedder behaviour

**Given** a new module `Familiar.Providers.OpenAIEmbedder` implementing `Familiar.Knowledge.Embedder`
**When** `embed/1` is called with a non-empty string
**Then** it POSTs to `<base_url>/embeddings` with body `%{model: <embedding_model>, input: <text>}`, `Authorization: Bearer <api_key>` header, and parses `data[0].embedding` from a 200 response.
**And** returns `{:ok, [float()]}` on success or `{:error, {:provider_unavailable, %{provider: :openai_compatible, reason: ...}}}` on any HTTP/transport/parse failure.
**And** returns `{:error, {:provider_unavailable, %{...reason: :missing_api_key}}}` when no API key is configured.
**And** returns `{:error, {:provider_unavailable, %{...reason: :missing_embedding_model}}}` when no embedding_model is configured.
**And** the adapter resolves config in this order: opts > env vars (`FAMILIAR_API_KEY`, `FAMILIAR_BASE_URL`, `FAMILIAR_EMBEDDING_MODEL`) > config.toml default provider > application config > built-in defaults — mirroring the precedence in `OpenAICompatibleAdapter`.

### AC2: Migration recreates the sqlite-vec table at 1536 dimensions

**Given** a new Ecto migration timestamped after the existing `knowledge_entries` migration
**When** `mix ecto.migrate` runs
**Then** the existing `knowledge_entry_embeddings` virtual table is dropped and recreated as `vec0(entry_id integer primary key, embedding float[1536])`.
**And** the migration's `down/0` recreates the original 768-dimension table so `mix ecto.rollback` works for development.
**And** a new singleton table `knowledge_embedding_metadata` is created with columns: `id` (default 1, primary key), `model_name` (string, nullable), `dimensions` (integer, nullable), `inserted_at`, `updated_at`. Single-row enforced via primary-key check or upsert pattern in the context module.
**And** the migration runs cleanly on a fresh database.
**And** running it on an existing database (with no embeddings — verified by the dev DB state at branch HEAD) drops and recreates without error.

### AC3: Knowledge module reads dimension from a single source of truth

**Given** the `@expected_embedding_dimensions 768` constant in `lib/familiar/knowledge/knowledge.ex:516`
**When** the constant is replaced
**Then** dimensions come from a new function `Familiar.Knowledge.embedding_dimensions/0` that reads `Application.get_env(:familiar, :embedding_dimensions, 1536)` so tests and config can override it.
**And** `insert_embedding/2` rejects vectors of the wrong length with `{:error, {:storage_failed, %{reason: :dimension_mismatch, expected: ..., got: ...}}}` (existing behavior, just sourced from the new function).
**And** `config/test.exs` sets `config :familiar, :embedding_dimensions, 1536` so test mocks generate 1536-length vectors going forward.

### AC4: `Familiar.Knowledge.EmbeddingMetadata` context tracks the active model

**Given** a new module `Familiar.Knowledge.EmbeddingMetadata` (file: `lib/familiar/knowledge/embedding_metadata.ex` + schema in same directory)
**When** the API is exercised
**Then** it exposes:
- `get/0 :: () :: {:ok, %{model_name: String.t() | nil, dimensions: integer() | nil}}` — fetches the singleton row, creating an empty one if missing
- `set/2 :: (model_name :: String.t(), dimensions :: integer()) :: {:ok, map()}` — upserts the singleton row
- `current_model/0 :: () :: String.t() | nil` — convenience accessor
- `check_drift/1 :: (configured_model :: String.t() | nil) :: :ok | {:warning, :model_changed | :model_unset, %{stored: ..., configured: ...}}`

**And** the schema has its own `@type t :: %__MODULE__{}` and a changeset that validates `dimensions > 0`.

**And** when a new entry is stored via `Knowledge.store_with_embedding/1` and the metadata row has `model_name == nil`, the model is recorded automatically (using the configured model name from `Familiar.Config` or runtime.exs).

### AC5: Startup drift detection warns the user

**Given** the application boots with a configured `embedding_model` that differs from the metadata row's `model_name`
**When** `Familiar.Application.start/2` runs (or another suitable single-shot place — see Dev Notes)
**Then** a `Logger.warning/1` is emitted: `[Familiar] Embedding model changed: stored=<X> configured=<Y>. Run \`fam context --reindex\` to re-embed knowledge entries with the new model. Search results will be inaccurate until reindex completes.`
**And** the warning fires only ONCE per process lifetime (no log spam).
**And** when the metadata is empty AND there are zero knowledge entries, no warning fires (clean state).
**And** when the metadata is empty AND there ARE entries, the same warning fires with `stored=unset`.
**And** the warning does NOT prevent startup or block any operation — it is informational only.

### AC6: `Familiar.Knowledge.reindex_embeddings/1` re-embeds every entry

**Given** a new public function `Knowledge.reindex_embeddings(opts \\ [])`
**When** called
**Then** it iterates every row in `knowledge_entries`, calls `Familiar.Providers.embed/1` on each entry's text, and atomically replaces the existing vector in `knowledge_entry_embeddings` for that entry.
**And** it accepts an `:on_progress` callback (`(processed :: integer(), total :: integer()) :: any()`) invoked after each successful re-embed, so the CLI can stream progress.
**And** on success it updates `EmbeddingMetadata` with the configured model name and dimensions, then returns `{:ok, %{processed: integer(), failed: integer(), errors: [{entry_id, reason}]}}`.
**And** failures on individual entries do NOT abort the entire reindex — they are accumulated into the `errors` list and the function still updates metadata if at least one entry succeeded.
**And** on a fully empty knowledge store (`processed: 0, failed: 0`), metadata is still updated to the configured model so the next entry inserted matches.

### AC7: `fam context --reindex` CLI command

**Given** `fam context --reindex`
**When** invoked
**Then** the CLI calls `Knowledge.reindex_embeddings/1` with an `:on_progress` callback that prints `Re-embedding entry N/M...` to stderr at most once per second (or once per N entries — pick a sensible throttle).
**And** the final result is rendered as `{:ok, %{reindexed: count, failed: count, errors: [...]}}` in JSON mode.
**And** the text mode output is `Reindexed N entries (M failed). Embedding model is now <model>.` plus a list of any errors.
**And** an existing `--refresh` invocation (`fam context --refresh`) is unchanged.
**And** `--reindex` and `--refresh` cannot be combined; passing both returns `{:error, {:usage_error, %{message: ...}}}`.

### AC8: Runtime config wires `OpenAIEmbedder` automatically

**Given** `provider.type == "openai_compatible"` AND `provider.embedding_model` is set in config.toml (or `FAMILIAR_EMBEDDING_MODEL` env var)
**When** `config/runtime.exs` runs
**Then** `Familiar.Knowledge.Embedder` is configured to `Familiar.Providers.OpenAIEmbedder` instead of `StubEmbedder`.
**And** when `embedding_model` is missing, `StubEmbedder` is still used (existing fallback).
**And** the existing Ollama path is unchanged.

### AC9: Generator emits the new default in `fam init` config.toml

**Given** `Familiar.Config.Generator.generate_default/2`
**When** it builds the OpenRouter section
**Then** `embedding_model = "openai/text-embedding-3-small"` is included on the active OpenRouter block.
**And** the comments above the providers section call out that switching the embedding model later requires `fam context --reindex`.

### AC10: Test baseline + flake check

**Given** all changes
**When** the suite runs
**Then** `mix test`, `mix credo --strict`, `mix format --check-formatted`, `mix dialyzer` all pass with 0 issues.
**And** the suite grows from 1124 → ~1124 + new tests with 0 failures.
**And** existing tests that hard-code 768-length vectors are updated to use a shared `deterministic_vector/2` helper (currently 6 copies in 6 test files) that reads `Familiar.Knowledge.embedding_dimensions/0`. Move that helper into `test/support/embedding_helpers.ex`.
**And** the new test file(s) survive **10 consecutive `mix test --seed $RANDOM`** runs with zero failures (per `feedback_flaky_tests_zero_tolerance`).

## Dev Notes

### Scope boundaries

**In scope:**
- New `OpenAIEmbedder` adapter for the OpenAI-compatible `/embeddings` endpoint
- Migration to 1536 dimensions
- Model-change detection metadata + startup warning
- `fam context --reindex` CLI command
- Shared test helper for embedding dimensions
- `StubEmbedder` honoring the configured dimension

**Out of scope:**
- Per-call dimension auto-discovery (running multiple models with different dimensions in the same DB)
- Online schema migration when the user switches models with a different dimension (out-of-band manual migration is the answer)
- Streaming embeddings or batch endpoints (`/v1/embeddings` supports `input: [...]` for batches but the existing `Embedder` behaviour is single-text; batching is a future optimization)
- Cost tracking, token counting, retry/backoff (the chat adapter doesn't have these either; consistency wins)
- Migration of existing data — the project memory confirms there is no production data to migrate
- Extending the `Embedder` behaviour with new callbacks — keep `embed/1` as the only callback

### Boundary structure

`Familiar.Providers.OpenAIEmbedder` is a member of the existing `Familiar.Providers` boundary (no `use Boundary` of its own). Same pattern as `OllamaEmbedder` and `OpenAICompatibleAdapter`. It does NOT need to be added to the boundary exports because it is only referenced via runtime config (`config :familiar, Familiar.Knowledge.Embedder, ...`), not aliased from outside.

`Familiar.Knowledge.EmbeddingMetadata` is a member of the existing `Familiar.Knowledge` boundary. Same pattern as `Familiar.Knowledge.Entry`, `Familiar.Knowledge.Freshness`, etc. No new boundary declaration.

### Drift detection placement

The first plausible place to call `check_embedding_drift/0` is `Familiar.Application.start/2` after the supervision tree is up. Wrap it in `Task.start/1` so it doesn't block boot if the Repo is slow. Use `Process.put`-style sentinel OR (better) `:persistent_term.put/2` to avoid re-warning across multiple `Application.start/2` calls (e.g., test reloading). Skip the check entirely when `Application.get_env(:familiar, :skip_embedding_drift_check, false)` is true — set this in `test.exs`.

**Do not** put the check in CLI bootstrap (`Familiar.CLI.Main.bootstrap/0`) because that would skip it for the daemon path. Application start is the single source of truth.

### Reindex transaction shape

Use `Repo.transaction(fn -> Repo.stream(Entry) |> Enum.reduce(...) end)` to keep memory bounded. `Repo.stream/2` requires being inside a transaction. The `:on_progress` callback can be invoked outside the transaction by accumulating progress in the reducer state and emitting after each entry. Since each entry update writes to the vec0 table (separate from `knowledge_entries`), the transaction provides consistency for the metadata update at the end.

If transactional streaming complicates the failure handling, the simpler alternative is to `Repo.all(Entry)` and iterate in process memory — knowledge stores in this codebase are small (hundreds of entries, not millions). **Pick the simpler alternative for the MVP** unless dev finds a reason to stream.

### Why `embedding_dimensions/0` reads from `Application` config

A function call instead of a module attribute is the cheapest way to make the dimension configurable per-test and per-environment without a recompile. 1536 is the production default; tests use the same 1536. Alt-dim providers (Gemini's 768, Cohere's 384) can be supported in a future story by changing the config and running a fresh migration.

### Reuse `OpenAICompatibleAdapter` config helpers

The chat adapter (`lib/familiar/providers/openai_compatible_adapter.ex`) already implements env > config.toml > app config > defaults precedence for `api_key`, `base_url`, and `chat_model`. Mirror that pattern in `OpenAIEmbedder` — extract the shared resolution logic into a small private module if you find yourself copy-pasting more than ~10 lines. The `Familiar.Daemon.Paths` alias is already wired into the chat adapter for `config_path()`, reuse it.

### Pricing context (informational, not enforced)

`openai/text-embedding-3-small` via OpenRouter is **$0.02/M tokens** as of the spec memory. Re-embedding a knowledge store of ~10k entries (typical project after a year of use) at ~200 tokens per entry costs ~$0.04. The startup drift warning is therefore safe — even if the user runs `--reindex` "by accident" the bill is negligible. Document this in the warning text so the user isn't scared.

### Flake prevention checklist (from `feedback_flaky_tests_zero_tolerance`)

- New tests touching the repo MUST be `async: false`
- No `Process.sleep/1` sync points; use `GenServer.call/2` or `assert_receive` (the only async work in this story is the `Task.start/1` drift check, which writes a single log line — check for it via `ExUnit.CaptureLog` against a *specific phrase*, not `assert log == ""`)
- No assert-on-empty-log assertions
- Time-based assertions: avoid them entirely; the metadata model uses `inserted_at`/`updated_at` timestamps only for ordering, not for behavior
- Stress test rule: 10 consecutive passes on `mix test --seed $RANDOM` before marking any task complete

### Open questions to resolve before dev-story (optional)

- **Streaming vs `Repo.all`** for reindex — leaning toward `Repo.all` for MVP simplicity. Decision is dev's call.
- **Batch embedding** — the OpenAI `/embeddings` endpoint accepts `input` as either a string or a list of strings. Batching could speed reindex by ~10x. **Decision: out of scope** — batch first improvement when reindex actually feels slow on a real corpus.
- **Drift warning text** — proposed wording is in AC5. Tweak as needed; the only requirement is that the warning includes the stored model, configured model, and the literal `fam context --reindex` command string.
