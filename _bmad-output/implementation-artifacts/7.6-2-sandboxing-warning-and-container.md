# Story 7.6-2: Sandboxing Warning + Reference Container

Status: done

## Story

As a Familiar user,
I want a prominent warning about execution risk plus a copy-pasteable container recipe,
So that I understand Familiar is running untrusted LLM output on my machine and I know exactly how to sandbox it.

## Acceptance Criteria

1. **AC1: Help text updated.** `fam --help` includes a one-line security notice pointing at `docs/sandboxing.md`.

2. **AC2: Init success warning.** The `fam init` success output (both auto-init via `ensure_project_initialized/0` and explicit `fam init` via `text_formatter("init")`) includes a clearly-marked security notice after the initialization summary. The notice names the risk and points at `docs/sandboxing.md`.

3. **AC3: README security warning.** The `README.md` top section (above installation) contains a clearly-marked security warning. The existing `## Sandboxing` section is retained and enhanced with the reference Docker invocation from AC4.

4. **AC4: Reference Dockerfile.** A `familiar/Dockerfile` provides a multi-stage build: `hexpm/elixir` builder stage compiling the escript, `debian:bookworm-slim` runtime stage installing it to `/usr/local/bin/fam`. Runs as non-root user. `FAMILIAR_PROJECT_DIR=/workspace`. `CMD ["fam", "help"]`. The Dockerfile builds successfully (`docker build -t familiar .`).

5. **AC5: Docker Compose file.** A `familiar/docker-compose.yml` wires the workspace mount (`./:/workspace`), LLM provider env vars (`FAMILIAR_API_KEY`, `FAMILIAR_BASE_URL`, `FAMILIAR_CHAT_MODEL`, `FAMILIAR_PROVIDER`), and network access. Quick-start becomes `docker compose run --rm familiar fam init`.

6. **AC6: Sandboxing doc.** A `familiar/docs/sandboxing.md` covers: (a) why no runtime safety — the honest explanation, (b) recommended Docker invocation with `--network`, `--read-only`, `--cap-drop` flags, (c) Podman invocation, (d) ephemeral VM options (Firecracker, QEMU), (e) "what a sandbox does NOT protect you from" section (data exfiltration, prompt injection, provider data policies).

7. **AC7: Clean toolchain.** `mix compile --warnings-as-errors`, `mix format --check-formatted`, `mix credo --strict`, `mix test`, `mix dialyzer` all pass.

8. **AC8: Tests.** Updated tests for `fam --help` (includes security notice). Updated init output tests if they exist.

9. **AC9: Stress-tested.** Every touched test file passes 50 consecutive runs with no flakes.

### Previous Dockerfile (deleted in 7.5-3)

The old Dockerfile is in git history at `de237f8~1:familiar/Dockerfile`. It was a single-stage build using `elixir:1.19.5-otp-28-slim`. The new one should be multi-stage (builder + runtime) for a smaller image. Key differences from the old one:

- Old: single-stage, ~700MB image, boots full OTP app per command (~5s cold start)
- New: multi-stage, escript-based, small runtime image, instant CLI execution
- Old: `WORKDIR /app`, `CMD ["sh", "/app/bin/cli_wrapper.sh"]`
- New: `WORKDIR /workspace`, `CMD ["fam", "help"]`, escript at `/usr/local/bin/fam`

### Warning text content

The warning should explicitly name these risks (from epic scope):
1. LLM generates tool calls
2. Tool calls include `write_file`, `delete_file`, `run_command`
3. The LLM is an untrusted actor
4. Running outside a container puts host filesystem and network at risk

Keep the tone honest and direct — no euphemisms. Reference `docs/sandboxing.md` for comprehensive options.

### Sandboxing doc structure

From epic scope, `docs/sandboxing.md` should cover:
1. Why no runtime safety — honest explanation (name-pattern matching = theater)
2. Recommended Docker invocation (`--network none`, `--read-only`, `--cap-drop ALL`)
3. Podman invocation (rootless, no daemon)
4. Ephemeral VM (Firecracker, QEMU) for paranoid workflows
5. What a sandbox does NOT protect you from (data exfiltration via API, prompt injection stealing env var secrets, provider data policies)
