# Story 7.5-6: Persist and Resume Workflow Runs

Status: done

## Story

As a user running a long workflow like `fam plan` or `fam do`,
I want Familiar to persist each completed step's output and let me resume an interrupted workflow from the next step instead of starting over,
so that a timeout, a `Ctrl+C`, or a provider error doesn't throw away an hour of planning work.

## Context

The user's exact words during Epic 7.5 manual testing: *"If we have a timeout we better be able to resume the workflow. [...] It's acceptable for now, but let's promote it to a story."*

Today, `Familiar.Execution.WorkflowRunner` is a GenServer whose state (`workflow`, `initial_context`, `current_step_index`, `step_results`, `status`) lives only in memory. When the blocking caller in `run_workflow_parsed/3 → await_completion/3` hits the 30-minute deadline, it returns `{:error, {:workflow_timeout, :deadline_exceeded}}` and the runner GenServer eventually dies with it. Every completed step's output is lost. Same story for `Ctrl+C`, a provider 5xx, or any `{:DOWN, ...}` on the agent process. This is the rude `:deadline_exceeded` message the user hit in Epic 7.5 testing.

Parallel but unrelated system: `fam plan --resume` already exists (`cli/main.ex:277-283, 804-846`). That command uses `Familiar.Conversations` (conversations + conversation_messages tables) to replay a multi-turn chat transcript as *context* for a fresh run of the `feature-planning` workflow from step 1. It does not preserve step-level progress. That mechanism is independent of this story and should be left alone — we are adding real workflow-level checkpointing, not changing conversation replay.

**Scope of "resume" in this story (MVP):**

- On every step transition, persist `current_step_index` and `step_results` to a new `workflow_runs` table.
- On workflow start, create a row with `status=running`.
- On success → `status=completed`. On step failure / agent crash → `status=failed`, `last_error` recorded. On caller timeout → the GenServer keeps running and just keeps checkpointing; caller-side interruption does NOT change DB status. The next resume invocation re-reads the row and continues from `current_step_index`.
- Resume means **restart the next incomplete step fresh** with the previously-completed steps' outputs visible in `step_results`. It does **not** mean restoring the in-flight agent's conversation mid-tool-call. That is out of scope.

## Acceptance Criteria

### AC1: Workflow Run Persistence Schema

**Given** a new Ecto migration
**When** it is run
**Then** a `workflow_runs` table exists with columns:
- `id` (primary key)
- `name` (string, not null — e.g. `"feature-planning"`)
- `workflow_path` (string, nullable — absolute path to source .md file, for resume parse)
- `scope` (string, not null, default `"workflow"`)
- `status` (string, not null — one of `running`, `completed`, `failed`)
- `current_step_index` (integer, not null, default `0`)
- `step_results` (text, not null, default `"[]"` — JSON-encoded list of `%{"step" => name, "output" => content}`)
- `initial_context` (text, nullable — JSON-encoded map of the initial caller context)
- `last_error` (text, nullable — inspect-formatted error reason on failure)
- `inserted_at`, `updated_at` (utc_datetime)

**And** indexes exist on `status` and on `inserted_at`.

### AC2: `Familiar.Execution.WorkflowRuns` Context Module

**Given** a new `Familiar.Execution.WorkflowRuns` module with an Ecto schema (`Familiar.Execution.WorkflowRuns.Run`)
**When** the API is exercised
**Then** it exposes:
- `create/3 :: (name :: String.t(), opts :: keyword()) :: {:ok, Run.t()} | {:error, term()}` — inserts with `status=running`, `current_step_index=0`, `step_results=[]`, and serialized `initial_context`.
- `get/1 :: (id :: integer()) :: {:ok, Run.t()} | {:error, {:workflow_run_not_found, map()}}`
- `checkpoint/3 :: (id :: integer(), new_index :: non_neg_integer(), step_results :: [map()]) :: {:ok, Run.t()} | {:error, term()}` — atomically writes both fields + `updated_at`.
- `complete/1 :: (id :: integer()) :: {:ok, Run.t()}` — sets `status=completed`.
- `fail/2 :: (id :: integer(), reason :: term()) :: {:ok, Run.t()}` — sets `status=failed`, records `inspect(reason, pretty: true)` in `last_error`.
- `list/1 :: (opts :: keyword()) :: {:ok, [Run.t()]}` — supports `status:` and `scope:` filters, newest-first, default `limit: 50`.
- `latest_resumable/1 :: (opts :: keyword()) :: {:ok, Run.t()} | {:error, {:no_resumable_workflow, map()}}` — most recent row where `status in ["running", "failed"]`, filterable by `scope:`.

**And** `step_results` and `initial_context` are transparently encoded/decoded via an Ecto custom type or changeset callbacks (prefer a dedicated `Familiar.Execution.WorkflowRuns.JSONField` type module used by both fields — keep it tiny, no dependencies outside `Jason`).

### AC3: `WorkflowRunner` Persists State on Every Transition

**Given** a workflow started via `run_workflow/3` or `run_workflow_parsed/3`
**When** the runner transitions state
**Then** persistence happens at exactly these points:
1. **On start** — `WorkflowRunner.init/1` creates a `workflow_runs` row and stores its `run_id` in GenServer state. (Must use `{:ok, state, {:continue, :persist_start}}` so the insert runs outside the caller's `start_link` timeout window.)
2. **After each successful step** — right after appending to `step_results` in `handle_step_success/2`, call `WorkflowRuns.checkpoint(run_id, new_index, new_results)`.
3. **On workflow completion** — in `complete_workflow/1`, call `WorkflowRuns.complete(run_id)` before notifying the caller.
4. **On workflow failure** — in `fail_workflow/2`, call `WorkflowRuns.fail(run_id, reason)` before notifying the caller.

**And** persistence errors do NOT kill the workflow: wrap each call with `case/rescue` and log a warning (`Logger.warning("[WorkflowRunner] Failed to persist checkpoint: #{inspect(reason)}")`). A DB outage degrades to in-memory-only behavior, matching pre-story semantics.

**And** when the caller's `await_completion/3` deadline expires, the runner GenServer is NOT killed by this story's changes — the existing behavior (caller returns timeout, GenServer keeps running until it naturally completes or fails) is preserved. The checkpoints are what enable resume; the timeout message just becomes non-destructive.

### AC4: `WorkflowRunner.resume_workflow/2`

**Given** a new public function `WorkflowRunner.resume_workflow(run_id, opts \\ [])`
**When** called with a valid `run_id` whose row has `status in ["running", "failed"]`
**Then** it:
1. Loads the row via `WorkflowRuns.get/1`; returns `{:error, {:workflow_run_not_found, _}}` if missing.
2. Returns `{:error, {:workflow_already_completed, %{id: run_id}}}` if `status == "completed"`.
3. Re-parses the workflow from `workflow_path` (returns `{:error, {:workflow_path_missing, _}}` if `workflow_path` is `nil`, or propagates parse errors).
4. If `current_step_index >= length(workflow.steps)`, marks the run completed and returns `{:ok, %{steps: persisted_step_results}}` (edge case: run was fully executed but never marked completed — e.g., crash between final `checkpoint` and `complete`).
5. Otherwise, starts a new `WorkflowRunner` GenServer preloaded with `current_step_index`, `step_results`, `initial_context`, and the existing `run_id` (so subsequent checkpoints update the same row), then blocks on `await_completion/3` like `run_workflow_parsed/3`.

**And** `run_id` is threaded through `init/1` via a new `:run_id` opt — when present, `init/1` does NOT create a fresh row and instead reuses the passed id.

**And** a new opt `:resume_from` carrying `%{current_step_index: int, step_results: list}` lets `init/1` hydrate the state without re-querying (keeps the DB read inside `resume_workflow/2`, not in the GenServer's init).

### AC5: `fam workflows resume` CLI Command

**Given** `fam workflows resume [--id <integer>]`
**When** invoked without `--id`
**Then** the CLI calls `WorkflowRuns.latest_resumable/1` and resumes that run, printing `Resuming workflow run #<id>: <name> (step <i>/<total>)` before starting.
**And** returns `{:error, {:no_resumable_workflow, _}}` with a friendly message when no resumable run exists.

**Given** `fam workflows resume --id 42`
**When** invoked
**Then** it loads run #42 explicitly and resumes it.
**And** returns the same errors as `WorkflowRunner.resume_workflow/2` if the run is missing, completed, or its `workflow_path` is stale/missing.

**And** the command supports `--json` mode and uses the standard `Output.format/3` + `Output.exit_code/1` pattern as other CLI commands.

**And** the existing `fam workflows`, `fam workflows <name>`, and `fam workflows run <path>` commands are unchanged.

### AC6: `fam workflows list-runs` CLI Command

**Given** `fam workflows list-runs [--status running|completed|failed] [--scope <scope>] [--limit N]`
**When** invoked
**Then** it prints a table of runs with columns: `id`, `name`, `status`, `step`, `updated_at`, `last_error` (truncated to 60 chars).
**And** supports `--json` mode returning the raw list of run maps.
**And** the default `--limit` is `20`.

### AC7: Resume Propagates Step Results Into Step Context

**Given** a workflow with 3 steps `[a, b, c]` where the original run succeeded at `a` and `b` and failed at `c`
**When** the user runs `fam workflows resume --id <id>`
**Then** the new runner skips `a` and `b` (they are already in `step_results`).
**And** starts a fresh agent for step `c` whose `build_task_description/2` sees both `a` and `b` as "Previous step results" exactly as if the original run had reached `c` naturally.
**And** on `c` success, the persisted row's `step_results` contains all three entries and `status=completed`.

**Given** all changes
**When** `mix test`, `mix credo --strict`, `mix format --check-formatted`, and `mix dialyzer` run
**Then** 0 failures, 0 Credo issues, 0 Dialyzer errors, format clean.
**And** the new test file runs 10 consecutive seeds without a single failure (per `feedback_flaky_tests_zero_tolerance` — any flake is a P1 blocker, no `Process.sleep/1` sync points, assert on content not empty-log, etc.).
**And** the baseline grows to approximately 1083 + (new tests) pass count.

## Dev Notes

### Scope boundaries

**In scope:**
- Persistence of workflow run progress at step boundaries.
- Resume-from-next-incomplete-step with prior step outputs intact.
- CLI entry points: `fam workflows resume [--id <id>]`, `fam workflows list-runs`.
- Fail-soft persistence: DB errors log a warning and the runner keeps going.

**Out of scope:**
- Mid-step state restoration. If an agent was at tool-call #7 when the runner died, resume restarts that step from its opening prompt. Documented in story Context.
- Changing `fam plan --resume` behavior (it stays on the Conversations-replay path).
- Persisting agent conversation messages per step — that's Story 7.5-7 territory or later.
- Migration from old to new data — nothing persists today, there is no legacy data.
- Any UI/TUI changes to surface resume — post-Epic 9.

### Key files

- `lib/familiar/execution/workflow_runner.ex` — threading persistence through init, step transitions, completion/failure.
- `lib/familiar/execution/workflow_runs/*.ex` — new context module and schema.
- `lib/familiar/cli/main.ex` — CLI wiring for `workflows resume` and `workflows list-runs`.
- `priv/repo/migrations/<stamp>_create_workflow_runs.exs` — migration.
- `config/runtime.exs` — no changes needed (Repo is already configured for CLI mode after Story 7.5-3).

### Patterns to reuse

- `Familiar.Conversations` is the reference implementation for a harness-owned Ecto context module (`lib/familiar/conversations/conversations.ex:17-143`). Mirror its error-tuple shape, `Boundary` declaration, and test layout.
- `Familiar.Execution.WorkflowRunner.start_next_step/1` and `handle_step_success/2` are the hot paths — keep persistence calls *outside* the tight loop, only at boundaries.
- Use `GenServer.handle_continue/2` (pattern already established in `Familiar.Execution.ToolRegistry` and `Familiar.Execution.AgentProcess`) to defer the initial DB insert so it does not block the caller's `start_link`.
- For JSON-as-text in SQLite, follow the pattern used by `conversation_messages.tool_calls` (`migrations/20260403200000_create_conversations.exs:16` — `:text` column + default `"[]"`, application-layer encode/decode).

### Flake prevention checklist (from `feedback_flaky_tests_zero_tolerance`)

- **No `Process.sleep/1` sync points.** The test that asserts checkpoints exist after step 2 must sync on a GenServer.call or on `await_completion` returning — not a sleep.
- **No `assert log == ""` on `capture_log`.** If you need to assert that no warning fired, use `refute log =~ "specific phrase"`.
- **`async: false` for any test that writes to the repo** or mutates `Application.put_env(:familiar, :project_dir, ...)`.
- **Initialize time-tracked state to `nil`**, not `now - cooldown`, if you add any rate-limited logging. (N/A for this story unless you add throttling, but a good reminder.)
- **Stress-test rule: 10 consecutive passes on `mix test --seed $RANDOM`** before marking any task complete.

### Open questions (resolve before dev-story if possible)

- Do we need a `scope` concept on workflow_runs beyond the hardcoded default? `Conversations` has one for planning vs. default; `WorkflowRunner` already accepts `:scope` in opts. Store it, don't use it yet.
- Should `WorkflowRuns.latest_resumable/1` include `running` status or only `failed`? Including `running` matches the use case "I killed fam and want to pick back up" but can cause confusion if another runner is actively executing. **Decision: include both** — the resume path is explicit user action and the worst case is a noisy double-run that will itself fail fast. Document this in the CLI output.

### Completion Notes List

- **Persistence model:** `workflow_runs` table stores one row per run. Writes happen at three GenServer checkpoints — `handle_continue(:persist_start)` on init, `persist_checkpoint/3` after every step success, and `persist_complete/1` / `persist_fail/2` at termination.
- **Fail-soft guarantee:** Every `WorkflowRuns.*` call from the runner goes through `safe_call/1`, which catches `rescue`/`:exit` so DB outages log a warning and the workflow continues with `run_id = nil`. Pre-story in-memory semantics are preserved when the repo is unreachable.
- **Hot path untouched:** Persistence only runs at step boundaries, not inside tight loops. `start_next_step/1` has zero new DB traffic.
- **Chronological in DB, newest-first in memory:** The in-memory list remains `[head | tail]` for `O(1)` appends; the persistence layer does a single `Enum.reverse/1` on write and `resume_workflow/2` does the mirror on read. Consumers of the row (CLI, list-runs output) see steps in execution order.
- **Resume edge cases:**
  - `workflow_already_completed` — surfaced when `status == "completed"` OR when `current_step_index >= length(steps)` (the post-final-step crash case is auto-finalized as a side effect).
  - `workflow_path_missing` — surfaced when a run was created via `run_workflow_parsed/3` without the new `:workflow_path` opt.
  - `workflow_run_not_found` — surfaced for missing ids.
  - Successful resume reuses the existing `run_id` so subsequent checkpoints update the same row instead of creating a duplicate.
- **`run_workflow/3` entry-point change:** now calls `Path.expand(path)` and stashes it into `opts[:workflow_path]` before delegating to `run_workflow_parsed/3`. `run_workflow_parsed/3` is untouched structurally — the new path just flows through as a regular opt.
- **Initial context JSON safety:** Added `sanitize_context/1` that stringifies atom keys and inspects non-JSON-safe values before persistence. Necessary because existing callers pass `%{task: "..."}` and occasional test helpers include pids/refs.
- **Context hydration on resume:** JSON decode produces string keys; `atomize_context/1` converts the one known atom key (`"task" -> :task`) back so existing agents don't need changes. Other keys stay as strings.
- **`fam plan --resume` left alone:** The conversation-replay resume path in `cli/main.ex:804-846` was audited and confirmed untouched. Updated `help_text/0` to list both `fam workflows resume` and (implicitly, via existing text) `fam plan --resume` so the two coexist.
- **No boundary changes needed:** `Familiar.Execution.WorkflowRuns` is a new module *inside* the `Familiar.Execution` boundary (no `use Boundary` of its own — it is a boundary member like `ToolRegistry`). Added it to the `Familiar.Execution` exports list so callers outside the boundary can alias it.
- **Test baseline grew from 1083 → 1117 tests** (+34 new across 4 new test files). 10 consecutive stress runs with random seeds all pass. `mix credo --strict`, `mix format --check-formatted`, `mix dialyzer` all clean.

### Review Findings

- [x] [Review][Patch] `fam workflows list-runs` text output falls through to `inspect/1` instead of a fixed-width table — **Fixed 2026-04-10**: added `format_workflow_runs_table/1` and a `%{runs: runs}` clause in `text_formatter("workflows")`. Renders columns `ID NAME STATUS STEP UPDATED ERROR` with `String.pad_trailing/3`.
- [x] [Review][Patch] `--id <non-integer>` silently falls through to `latest_resumable` — **Fixed 2026-04-10**: `parse_args/1` now stashes the rejected switches under `flags[:_invalid]`. `run_workflows_resume/2` calls `reject_invalid_id/1` which surfaces `{:error, {:usage_error, %{message: "Invalid --id value: ..."}}}` before any context lookup.
- [x] [Review][Patch] Flaky `list/1` ordering on same-second `inserted_at` AND inconsistent ordering between `list/1` and `latest_resumable/1` — **Fixed 2026-04-10**: both queries now `order_by: [desc: r.inserted_at, desc: r.id]`. The `desc: r.id` secondary sort makes ordering deterministic even when `:utc_datetime` second-precision rows tie.
- [x] [Review][Patch] `JSONField.load` decode failure returns `{:ok, nil}` against schema default `[]` — **Fixed 2026-04-10**: now returns `{:ok, []}` on decode failure with a clarifying log message. Both fields backed by this type treat an empty list as the safe degraded state.
- [x] [Review][Patch] `WorkflowRuns.complete/1` does not clear `last_error` — **Fixed 2026-04-10**: changeset now sets `last_error: nil` along with `status: "completed"`. Verified via new test `complete/1 clears last_error`.
