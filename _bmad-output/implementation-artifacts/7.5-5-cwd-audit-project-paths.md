# Story 7.5-5: Replace `File.cwd!()` With `Paths.project_dir()` in Project-Scoped Modules

Status: done

## Story

As a developer running `fam` from any directory,
I want roles, skills, workflows, safety sandboxing, and provider config to always resolve against the user's project directory (via `FAMILIAR_PROJECT_DIR`),
so that CLI commands don't silently fall back to the Familiar source directory and load/enforce the wrong files.

## Context

`Familiar.Daemon.Paths.project_dir/0` already centralizes project directory resolution with the correct priority: `Application.get_env(:familiar, :project_dir) || System.get_env("FAMILIAR_PROJECT_DIR") || File.cwd!()` (`familiar/lib/familiar/daemon/paths.ex:10-14`). The CLI wrapper (`bin/fam`) sets `FAMILIAR_PROJECT_DIR` to the user's current directory before invoking `mix run` from the Familiar source tree, so `File.cwd!()` is almost always wrong in CLI mode.

Several production modules still default to `File.cwd!()` directly, which causes:

1. **Roles/skills loaded from the Familiar source dir** — when `familiar_dir:` is not explicitly passed (CLI command paths that go straight into `Familiar.Roles`), we look at `<familiar-source>/.familiar/roles/`, not the user's `.familiar/roles/`.
2. **Workflows listed from the wrong directory** — `WorkflowRunner.list_workflows/1` with no opts hits `<familiar-source>/.familiar/workflows/`.
3. **Safety extension sandbox rooted at the Familiar source dir** — path sandboxing, `.git/` protection, and command allow-listing check containment against the wrong root, letting tools write outside the user's project (sandbox is too permissive/restrictive depending on shape).
4. **Provider config.toml loaded from the wrong directory** — `load_project_config/0` in `OpenAICompatibleAdapter` also duplicates the same `Application.get_env || System.get_env || File.cwd!()` chain instead of calling `Paths.project_dir/0`.

Epic 7.5 manual testing uncovered each of these as real UX bugs (see `.claude/.../memory/project_cwd_audit.md`). `file_watcher.ex` and `tools.ex` file/command tools were already fixed in previous stories; this story cleans up the rest.

This is a pure refactor: the new behavior matches what every call site already tries to do in production, and all existing tests inject `familiar_dir:`/`project_dir:` explicitly so tests remain unaffected. No AC touches the CLI wrapper or runtime config — those are already correct.

## Dev Notes

### Key modules

- **`Familiar.Daemon.Paths`** (`lib/familiar/daemon/paths.ex`) — canonical source of project directory resolution. This story threads it through every remaining call site.
- **Module boundaries** — `Familiar.Daemon` has no `use Boundary` declaration, so it is unchecked by the Boundary library and may be referenced from any module. `Familiar.Execution.FileWatcher` already imports `Familiar.Daemon.Paths` (`lib/familiar/execution/file_watcher.ex:35`) without a boundary exception. No boundary changes should be required — but if `mix compile --warnings-as-errors` flags a boundary violation for any of the edited modules, stop and report rather than silently adding Boundary exports.

### Why this matters

The CLI wrapper sets `FAMILIAR_PROJECT_DIR` to the user's cwd, but the `mix run` inside the wrapper changes cwd to the Familiar source tree. Any code that calls `File.cwd!()` from that point on gets the wrong answer. `Paths.project_dir/0` is the single source of truth; every default must go through it.

### Not in scope

- Removing `File.cwd!()` from test helpers (tests run from the repo root, so `File.cwd!()` is correct there).
- Changing `cli/main.ex:33` — the bootstrap explicitly reads `FAMILIAR_PROJECT_DIR` before `Paths` is used; it's correct as-is.
- Fixing `file_watcher.ex` runtime behavior — already correct, only the docstring needs a cosmetic update (part of Task 6).
- Introducing a Boundary change for the broader codebase — only add `Familiar.Daemon` to the minimum set of modules that now need it.

### Testing standards

- Tests use `ExUnit`. Most existing tests create a `tmp_dir` via `@tag :tmp_dir` and pass it explicitly via `familiar_dir:` or `project_dir:`. The new default-behavior tests should use `Application.put_env/3` + `on_exit` to isolate global state. Do **not** use `System.put_env/2` for `FAMILIAR_PROJECT_DIR` in tests — it leaks across parallel tests. `Application.put_env` is also shared, so mark such tests `async: false`.
- The pre-commit hook runs compile (warnings-as-errors), format, credo strict, sobelow, test, dialyzer — all six must pass before commit.
