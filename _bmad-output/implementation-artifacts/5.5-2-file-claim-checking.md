# Story 5.5.2: File-Level Claim Checking

Status: done

## Story

As the harness runtime,
I want file write/delete operations to check for existing claims before proceeding,
so that concurrent agents cannot silently overwrite each other's in-progress file changes.

## Context

The `Files.write/3` function logs a "pending" transaction via `insert_transaction/1` then proceeds with the write. The unique constraint on `[:task_id, :file_path]` prevents the same task from writing the same file twice, but does NOT prevent different tasks from claiming the same path. Two concurrent agents can both insert pending transactions for the same file — the second write silently overwrites the first agent's work.

This was identified in the Epic 5 retrospective as a "Must" action item for Epic 5.5.

## Acceptance Criteria

### AC1: Write Rejects When File Is Already Claimed by Another Task

**Given** Agent A has a pending transaction for `/project/foo.ex`
**When** Agent B calls `Files.write("/project/foo.ex", content, "task_b")`
**Then** the write returns `{:error, {:file_claimed, %{path: path, owner: "task_a"}}}`
**And** no transaction is inserted for Agent B
**And** no file is written to disk

### AC2: Delete Rejects When File Is Already Claimed

**Given** Agent A has a pending transaction for `/project/foo.ex`
**When** Agent B calls `Files.delete("/project/foo.ex", "task_b")`
**Then** the delete returns `{:error, {:file_claimed, %{path: path, owner: "task_a"}}}`

### AC3: Same Task Can Re-Write Its Own Claimed File

**Given** Agent A has a pending transaction for `/project/foo.ex`
**When** Agent A calls `Files.write("/project/foo.ex", new_content, "task_a")`
**Then** the existing transaction is updated (or the unique constraint handles it as before)
**And** the write proceeds normally

### AC4: Completed/Rolled-Back Transactions Don't Block

**Given** a transaction for `/project/foo.ex` with status "completed"
**When** any agent calls `Files.write("/project/foo.ex", content, "task_b")`
**Then** the write proceeds normally (completed transactions are not active claims)

### AC5: Tool Layer Returns Meaningful Error

**Given** `write_file` tool is called and the file is claimed by another agent
**When** `Tools.write_file/2` receives `{:error, {:file_claimed, info}}`
**Then** it returns a structured error the LLM can understand (not a raw changeset error)

### AC6: Test Baseline

**Given** all changes are applied
**When** `mix test` runs
**Then** all existing tests pass with 0 failures, 0 regressions
**And** Credo strict passes with 0 issues

## Dev Notes

### Implementation: Single Query Check

The claim check is a simple Ecto query — no ETS, no GenServer, no distributed lock. SQLite serializes writes anyway, so the check-then-insert is atomic within a single Repo operation:

```elixir
defp check_claim(path, task_id) do
  case Repo.one(
    from t in Transaction,
      where: t.file_path == ^path and t.task_id != ^task_id
             and t.status in ["pending", "conflict"],
      select: t.task_id,
      limit: 1
  ) do
    nil -> :ok
    owner -> {:error, {:file_claimed, %{path: path, owner: owner}}}
  end
end
```

This is sufficient because:
- SQLite is single-writer — concurrent queries serialize at the database level
- The check and insert happen in the same process (no interleaving)
- Even if a race occurs (check passes, another agent inserts between check and our insert), the unique constraint on `[:task_id, :file_path]` catches it as a changeset error

### Note on Unique Constraint

The existing unique constraint is on `[:task_id, :file_path]` — this prevents the SAME task from writing the SAME file twice. It does NOT prevent different tasks from writing the same file. The claim check fills this gap.
