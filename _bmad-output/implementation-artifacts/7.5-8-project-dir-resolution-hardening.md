# Story 7.5-8: Project-Dir Resolution & Entry-Point Hardening

Status: done

## Story

As a Familiar user whose daemon, CLI, escript, and background jobs all need to agree on which directory is "the project",
I want a single explicit resolution path that supports cwd walk-up and fails loudly when it can't resolve,
so that processes spawned by editors, CI, launchd, systemd, or unfamiliar shells don't silently fall back to the wrong directory — and so that `fam` commands work from any subdirectory of a project the way `git` does.

## Context

Story 7.5-5 audited `File.cwd!/0` usage and routed most production modules through `Familiar.Daemon.Paths.project_dir/0` (see `familiar/lib/familiar/daemon/paths.ex:10-14`). That closed the bulk of the obvious holes but left the *entry-point* question unsolved: when a process first starts (daemon boot, escript invocation, CLI command, background job from a supervisor restart), what decides which project it belongs to?

Today the answer is `Application.get_env(:familiar, :project_dir) || System.get_env("FAMILIAR_PROJECT_DIR") || File.cwd!()`, with:

- **No walk-up.** If you `cd ~/myproject/lib/some/sub/dir` and run `fam`, bin/fam sets `FAMILIAR_PROJECT_DIR` to your current subdirectory. `.familiar/` lives at `~/myproject/.familiar`, so `Paths.familiar_dir/0` returns `~/myproject/lib/some/sub/dir/.familiar` — which doesn't exist. The daemon then auto-inits a NEW `.familiar/` in the subdirectory, orphaning the real project. Users hit this and have to `cd` to the project root every time.
- **No validation.** If `FAMILIAR_PROJECT_DIR` points at a non-existent directory, or at one that isn't a Familiar project, `Paths.project_dir/0` returns it anyway. Downstream code crashes with opaque file-not-found errors that don't tell the user where to fix their config.
- **Inconsistent error messages.** Every entry point that tries to resolve project_dir hits a different fallback path and emits a different error. There's no single place to fix it.
- **No debug surface.** When "why is Familiar looking at the wrong directory?" happens (and it does — this story exists because it already happens), the user has no way to ask Familiar what it thinks the project dir is and why.

**The fix:** introduce `Paths.resolve_project_dir/2` as the single precedence chain with cwd walk-up (matching how `git` resolves the repo root), add a companion `Paths.validate_familiar_project/1`, standardize the error atoms in `Familiar.CLI.Output`, add a `fam where` debug command, and route all entry points through the new resolver.

**Bootstrap constraint:** `familiar/lib/familiar/cli/main.ex:34` reads `FAMILIAR_PROJECT_DIR` directly before `Application.ensure_all_started(:familiar)` is called, because the initialization check has to run before the app starts. Similarly, `familiar/config/runtime.exs:21,41` reads the env var at config-load time, before any modules are available. These two call sites cannot use arbitrary Familiar modules — any code they call has to be self-contained. The new resolver MUST be callable from these sites without pulling in the full application, which means it can't touch `Repo`, `Logger` setup, extensions, or anything that assumes the app is running. It's a pure function over `explicit_arg + env_var + cwd + filesystem reads`.

**Why walk-up stops at filesystem root, not `$HOME`:** `$HOME` is seductive (faster, avoids climbing `/home/`, `/`) but is a footgun: a user whose project is `~/myproject` would have `~/myproject/.familiar`, and walk-up from `~/myproject/lib` would find it before hitting `$HOME`. Good. But a user whose project is *at* `~/` (rare but legal) would have `~/.familiar`, and stopping at `$HOME` means we'd never check `~/` itself. Walk-up to `/` is simpler, matches `git`, and is fast enough (`.familiar/` check is a single `File.dir?` per ancestor, and typical directory depths are <10).

## Acceptance Criteria

### AC1: `Paths.resolve_project_dir/2` precedence chain

**Given** a new public function `Familiar.Daemon.Paths.resolve_project_dir/2` with signature `(explicit :: String.t() | nil, opts :: Keyword.t()) :: {:ok, String.t(), source :: atom() | {atom(), String.t()}} | {:error, {:project_dir_unresolvable, map()}}`
**When** called
**Then** it applies this precedence, short-circuiting on the first hit:

1. **`explicit`** (non-nil). Source = `:explicit`. Return `{:ok, Path.expand(explicit), :explicit}`.
2. **`FAMILIAR_PROJECT_DIR`** env var (non-nil, non-empty). Source = `:env`. Return `{:ok, Path.expand(env_value), :env}`.
3. **Cwd walk-up** — starting at `File.cwd!/0`, check each ancestor for a `.familiar/` subdir; stop at the filesystem root. If a match is found at ancestor `A`, return `{:ok, A, {:walk_up, A}}`.
4. **Cwd fallback** — only if `opts[:allow_cwd_fallback] == true` (default `false`). Return `{:ok, File.cwd!/0, :cwd_fallback}`. This is what `fam init` uses — it legitimately needs to resolve the dir even when no `.familiar/` exists.
5. **Hard error** — return `{:error, {:project_dir_unresolvable, %{cwd: File.cwd!/0, checked_env: boolean(), explicit: nil}}}`. `checked_env` is `true` when `FAMILIAR_PROJECT_DIR` was checked and found nil/empty.

**And** `opts` also accepts `:env` (override the env lookup, for tests) and `:cwd` (override the cwd lookup, for tests), both defaulting to real system calls — no global state mutation required in tests.

**And** all returned paths are absolute and expanded via `Path.expand/1`.

### AC2: Walk-up algorithm

**Given** `Paths.find_familiar_root/1` (private helper or exposed for testing — dev's choice)
**When** called with a starting directory
**Then** it walks up parent directories until either:
- a directory containing a `.familiar/` subdirectory is found → return `{:ok, that_directory}`
- the filesystem root `/` is reached without finding one → return `:not_found`

**And** the starting directory is normalized via `Path.expand/1` before walking (so `.` / `..` / trailing-slash components are collapsed). Note: `Path.expand/1` does NOT follow symlinks — walk-up is purely textual. Symlink-aware walk-up would require `File.read_link/1` + cycle detection and is out of scope for this story; the textual walk matches how `git rev-parse --show-toplevel` behaved in earlier git versions and is predictable.

**And** the walk never crosses the filesystem root (no infinite loops, no climbing past `/`).

**And** walk-up does NOT check the starting directory's parent first — it checks the starting directory itself, then ancestors. So running `fam` from the project root (which has `.familiar/`) correctly finds itself without climbing.

### AC3: `Paths.project_dir/0` gains walk-up + cwd fallback

**Given** the existing `Paths.project_dir/0` function
**When** called
**Then** it becomes a thin wrapper: `{:ok, dir, _source} = resolve_project_dir(nil, allow_cwd_fallback: true); dir`. Same return type as today (bare string), gains walk-up behavior silently for all existing callers. Never raises — `allow_cwd_fallback: true` guarantees a returned dir.

**And** a new companion `Paths.project_dir_or_error/0` returns `{:ok, dir, source} | {:error, reason}` for callers that want to surface the error (notably `fam where` and commands that require a real project).

**And** existing tests that call `Application.put_env(:familiar, :project_dir, ...)` continue to work — the explicit-arg path takes precedence when a test sets it via opts, and the env-var + walk-up paths are what changes.

**BUT** `Application.get_env(:familiar, :project_dir)` is no longer consulted by `resolve_project_dir/2` — the new resolver is pure over its arguments + env + cwd. To preserve the `Application.put_env`-based test override pattern from 7.5-5, `Paths.project_dir/0` reads `Application.get_env(:familiar, :project_dir)` and passes it as the `explicit` argument to `resolve_project_dir/2` when set. This keeps the existing test fixtures working unchanged.

### AC4: `Paths.validate_familiar_project/1`

**Given** a new function `Paths.validate_familiar_project/1` with signature `(dir :: String.t()) :: :ok | {:error, {:not_a_familiar_project, %{path: String.t()}}}`
**When** called
**Then** it returns `:ok` if `Path.join(dir, ".familiar")` is a directory, otherwise `{:error, {:not_a_familiar_project, %{path: dir}}}`.

**And** this function is called by entry points that require an initialized project (daemon boot, most CLI commands). `fam init` and `fam where` do NOT call it.

### AC5: Error messages in `Familiar.CLI.Output`

**Given** two new error atoms: `:project_dir_unresolvable` and `:not_a_familiar_project`
**When** `Familiar.CLI.Output.error_message/2` is called with either
**Then** it returns a friendly, actionable message that includes:

- **For `:project_dir_unresolvable`:**
  ```
  Could not determine the Familiar project directory.

  Checked:
    cwd:                   <actual cwd>
    FAMILIAR_PROJECT_DIR:  <value or "(unset)">
    walk-up:               no .familiar/ found in any ancestor of cwd

  Fix: either run `fam` from inside a Familiar project, set the env var
    export FAMILIAR_PROJECT_DIR=/path/to/project
  or pass the flag
    fam --project-dir /path/to/project <command>

  Run `fam where` from any shell to debug resolution.
  ```

- **For `:not_a_familiar_project`:**
  ```
  <path> exists but is not a Familiar project (no .familiar/ subdirectory).

  Fix: run `fam init` from inside that directory, or point Familiar elsewhere:
    export FAMILIAR_PROJECT_DIR=/path/to/real/project
    fam --project-dir /path/to/real/project <command>
  ```

**And** details maps carry enough structured data for JSON-mode output: `%{cwd: ..., env: ..., explicit: ...}` for `:project_dir_unresolvable`, `%{path: ...}` for `:not_a_familiar_project`.

**And** both error messages are emitted via `error_message/2`'s existing pattern (pure string builders, no IO).

### AC6: `fam where` debug command

**Given** a new CLI command `fam where`
**When** invoked
**Then** it prints a diagnostic dump of the current resolution state to stdout. In `:text` mode, output format:

```
project_dir:  /home/user/code/myapp
source:       walk-up (found .familiar/ at /home/user/code/myapp)
cwd:          /home/user/code/myapp/lib/myapp
env:          FAMILIAR_PROJECT_DIR = (unset)
explicit:     --project-dir (unset)
familiar_dir: /home/user/code/myapp/.familiar  [exists]
config:       /home/user/code/myapp/.familiar/config.toml  [exists]
daemon:       running (pid 12345, socket /home/user/code/myapp/.familiar/daemon.sock)
```

**And** in `:json` mode, it emits the same information as a structured data envelope with snake_case keys.

**And** when resolution *fails* (`:project_dir_unresolvable`), `fam where` still runs, prints what was checked, and exits with the normal non-zero error code. It is explicitly designed to be usable from a broken shell — that's its whole purpose.

**And** `fam where` works without the daemon running. Daemon status is shown as `stopped` (or `stale` if a stale PID file is found) — this diagnostic shouldn't require starting the daemon.

**And** `fam where` is a local command (like `fam version`) — it does NOT go through `run_with_daemon/2`.

2. **`lib/familiar/daemon/paths.ex:10-14`** — `project_dir/0` rewritten per AC3.
3. **`lib/familiar/execution/tools.ex:258-279`** — `project_cmd_opts/0` and `project_path/1` intentionally pass through when `FAMILIAR_PROJECT_DIR` is unset (test fixtures rely on this). **Left alone.** Documented in-line (already documented from 7.5-5).
4. **`lib/familiar/application.ex:138`** — `cli_mode?/0` reads `FAMILIAR_PROJECT_DIR` as a sentinel for "CLI invocation via bin/fam." **Left alone** — this is a mode flag, not a path resolution, and changing it risks test regressions in `skip_migrations?/0` and friends. ### AC8: `bin/fam` walk-up

**Given** `familiar/bin/fam` (the shell wrapper that escript-mode users invoke)
**When** a user runs `fam` from inside a subdirectory of a Familiar project
**Then** bin/fam walks up from `pwd` looking for a `.familiar/` directory in bash, and sets `FAMILIAR_PROJECT_DIR` to the found root (or to `pwd` if no root is found, matching today's behavior). This fix must happen in bash, not Elixir, because bin/fam already sets `FAMILIAR_PROJECT_DIR` unconditionally and by the time Elixir runs, the env var wins over the Elixir walk-up (per AC1 precedence).

**And** the bash walk-up respects the `FAMILIAR_PROJECT_DIR` env var if the user set it manually (i.e., `${FAMILIAR_PROJECT_DIR:-$(walk_up || pwd)}`).

**And** the bash implementation is a small, self-contained function (<15 lines) — no external tools beyond POSIX `test`/`dirname`/shell builtins.

**And** the bash walk-up and the Elixir walk-up (AC2) are behaviorally equivalent: both look for a `.familiar/` subdirectory in each ancestor, both stop at `/`, both start with the current directory itself before climbing.

### Review Findings

- [x] [Review][Defer] **`bin/fam` walk-up makes `fam init` for nested subprojects require a workaround** — `cd ~/work/big-project/sub && fam init` walks up to the outer project's `.familiar/`, reports "already initialized," and refuses. Reason for defer: matches git's nested-repo behavior; document `FAMILIAR_PROJECT_DIR=$(pwd) fam init` workaround instead.

- [x] [Review][Patch] **[HIGH] `bin/fam` walk-up can infinite-loop on MacOS bash 3.2 with a deleted cwd** — On MacOS default bash 3.2, `set -e` does not propagate into command substitution, so `dir=$(pwd)` can yield `""` when cwd was deleted. `dirname ""` returns `.`, then `dirname "."` returns `.`, and the `while [ "$dir" != "/" ]` loop never terminates. Fix: guard the loop with `[ -n "$dir" ] && [ "$dir" != "/" ] && [ "$dir" != "." ]` and validate `dir=$(pwd -P)` returned non-empty. [familiar/bin/fam:22-32]

- [x] [Review][Patch] **[MED] `fam where` can never surface `:project_dir_unresolvable` (AC6 violation)** — `build_where_diagnostic/2` calls `resolve_project_dir/2` with `allow_cwd_fallback: true`, so the error branch is unreachable and the `case` expression with a single `{:ok, ...}` clause is a time bomb. AC6 explicitly requires `fam where` to exit non-zero when resolution fails. Fix: call `resolve_project_dir/2` without fallback first; on `{:error, _}` return `{:error, {:project_dir_unresolvable, diag}}` with the partial diagnostic attached; on `{:ok, ...}` return `{:ok, diag}` as today. Add a test. [familiar/lib/familiar/cli/main.ex:1390-1396]

- [x] [Review][Patch] **[MED] `fam where` hint missing from init banner (AC10(b) violation)** — `ensure_project_initialized/0` / `run_init/1` emit init messages but never mention `fam where`. AC10(b) required this. Fix: add `IO.puts(:stderr, "[familiar] Run \`fam where\` anytime to debug project-dir resolution.")` to either the bootstrap banner or the init success output. [familiar/lib/familiar/cli/main.ex:73-93]

- [x] [Review][Patch] **[MED] `project_dir/0` match is a time bomb** — `{:ok, dir, _} = resolve_project_dir(explicit, allow_cwd_fallback: true)` asserts the never-error contract. Moduledoc promises "Never raises" but the match is load-bearing. ### Why the injected-opts pattern for tests

Per Story 7.5-5's review finding: `paths_test.exs` had to be flipped from `async: true` to `async: false` because it mutates `Application.put_env(:familiar, :project_dir, ...)`, racing with other async tests. For `resolve_project_dir/2`, the cleanest fix is to make env/cwd injectable via opts (default to real system calls). This means `paths_resolve_test.exs` can be `async: true` and run in parallel with everything else. Do NOT use `System.put_env("FAMILIAR_PROJECT_DIR", ...)` in tests — per the project memory `feedback_flaky_tests_zero_tolerance.md`, env-var mutation leaks across parallel tests and creates flakes.

### Walk-up algorithm pseudo-code

```elixir
defp find_familiar_root(start_dir) do
  dir = Path.expand(start_dir)

  cond do
    File.dir?(Path.join(dir, ".familiar")) -> {:ok, dir}
    dir == "/" -> :not_found
    true -> find_familiar_root(Path.dirname(dir))
  end
end
```

Recursive, bounded by filesystem depth (typically <20). `Path.expand/1` resolves symlinks at the entry point so we don't get confused by symlink loops at deeper levels (though a pathological symlink chain could still loop; if this becomes a real problem, add a max-depth counter — not scoped for MVP).

### Bootstrap constraint — what `cli/main.ex:34` can do

At line 34, `Application.ensure_all_started(:familiar)` has not yet been called, so:

- `Application.get_env(:familiar, :project_dir)` — **works** (compile-time config is loaded by BEAM before code execution); but no test or production config sets this at compile time, so it's always `nil` here.
- `Familiar.Daemon.Paths` — **safe to call** (pure module, no GenServer, no Repo).
- `System.get_env/1`, `File.cwd!/0`, `File.dir?/1`, `Path.*` — **safe**.
- `Logger` — avoid (Logger may not be configured for the target device; the existing code uses `IO.puts(:stderr, ...)`).
- `Repo`, extensions, `Familiar.Config.load/1` — **NOT safe** (app isn't started).

The new `resolve_project_dir/2` is pure and safe here. Updating `cli/main.ex:34` to call it is both correct and strictly better than the current inlined env+cwd check.

### `config/runtime.exs` is out of scope

`config/runtime.exs:21` reads `FAMILIAR_PROJECT_DIR` to set the Repo database path. It runs at config-load time — before any Familiar modules exist. Adding walk-up here would require reimplementing the walk in pure Elixir inside the config block. This is tractable (~10 lines) but out of scope for this story. Document it as a known gap: for the rare case where a user has `FAMILIAR_PROJECT_DIR` unset and invokes Familiar directly without bin/fam (which now does the bash walk-up), the Repo path won't get walk-up treatment. In practice this only affects people running the daemon via `mix phx.server` from a random directory — an edge case that pre-dates this story. If it becomes a real problem, a follow-up story can inline the walk-up into runtime.exs.
