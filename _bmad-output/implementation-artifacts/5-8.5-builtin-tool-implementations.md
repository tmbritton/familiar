**Given** the tool is registered

**And** returns `{:ok, %{content: "file contents..."}}` on success
**And** returns `{:error, {:file_error, %{path: ..., reason: ...}}}` on failure

**Given** the tool is registered
**When** an agent calls `:write_file` with `%{path: "lib/foo.ex", content: "..."}`
**Then** it delegates to the `FileSystem` behaviour port (`impl.write/2`)
**And** creates parent directories if needed
**And** returns `{:ok, %{path: "lib/foo.ex"}}` on success
**And** returns `{:error, reason}` on failure

### AC3: File Delete Tool (`:delete_file`)

**Given** the tool is registered

**Then** it delegates to the `FileSystem` behaviour port (`impl.delete/1`)

**And** returns `{:error, reason}` on failure

### AC4: List Files Tool (`:list_files`)

**Given** the tool is registered
**When** an agent calls `:list_files` with `%{path: "lib/"}`
**Then** it delegates to the `FileSystem` behaviour port (`impl.ls/1`)
**And** returns `{:ok, %{files: ["foo.ex", "bar.ex"]}}`
**And** returns `{:error, reason}` for invalid paths

### AC5: Search Files Tool (`:search_files`)

**Given** the tool is registered
**When** an agent calls `:search_files` with `%{path: "lib/", pattern: "defmodule"}`
**Then** it reads files in the directory and filters by content pattern
**And** returns `{:ok, %{matches: [%{path: "...", line: N, content: "..."}]}}`
**And** returns `{:ok, %{matches: []}}` for no matches

### AC6: Run Command Tool (`:run_command`)

**Given** the tool is registered
**When** an agent calls `:run_command` with `%{command: "mix test test/my_test.exs"}`
**Then** it parses the command string and delegates to the `Shell` behaviour port
**And** returns `{:ok, %{output: "...", exit_code: 0}}` on success
**And** returns `{:ok, %{output: "...", exit_code: N}}` for non-zero exit
**And** the safety extension's command allow-list is enforced via the hooks pipeline (not re-checked here)

### AC7: Spawn Agent Tool (`:spawn_agent`)

**Given** the tool is registered
**When** an agent calls `:spawn_agent` with `%{role: "coder", task: "implement auth"}`
**Then** it calls `AgentSupervisor.start_agent/1` with the role and task
**And** passes the calling agent as `:parent` so the child notifies on completion
**And** returns `{:ok, %{agent_id: "agent_N", pid: pid}}`
**And** returns `{:error, reason}` if the agent fails to start

### AC8: Monitor Agents Tool (`:monitor_agents`)

**Given** the tool is registered

**Then** it calls `AgentProcess.list_agents/0`
**And** returns `{:ok, %{agents: [%{pid: pid, agent_id: "agent_N"}]}}`

### AC9: Broadcast Status Tool (`:broadcast_status`)

## Dev Notes

### Architecture Constraints

- **Behaviour ports, not direct calls** â€” All file operations go through `Familiar.System.FileSystem`, shell through `Familiar.System.Shell`. This is the hexagonal architecture pattern established in Epic 2. Tools MUST NOT call `File.*` or `System.cmd` directly. [Source: architecture.md, config/test.exs mock config]
- **Safety already handled** â€” The safety extension runs as a `before_tool_call` alter hook at priority 1 before any tool function executes. Tool implementations do NOT re-check paths or commands â€” that's safety's job.
- **Args support atom and string keys** â€” LLM may send either. Use `Map.get(args, :key, Map.get(args, "key"))` pattern (established in Stories 5.6/5.7).

### Behaviour Port Resolution

```elixir
# Pattern used across the codebase:
defp file_system, do: Application.get_env(:familiar, Familiar.System.FileSystem)
defp shell, do: Application.get_env(:familiar, Familiar.System.Shell)

# Usage:
file_system().read(path)
shell().cmd(executable, args, [])
```

Config mapping (from `config.exs` and `test.exs`):
- Production: `FileSystem â†’ LocalFileSystem`, `Shell â†’ RealShell`
- Test: `FileSystem â†’ FileSystemMock`, `Shell â†’ ShellMock`

### Existing System Module Signatures

```elixir
# FileSystem behaviour:
read(path) :: {:ok, binary()} | {:error, {atom(), map()}}
write(path, content) :: :ok | {:error, {atom(), map()}}
delete(path) :: :ok | {:error, {atom(), map()}}
ls(path) :: {:ok, [String.t()]} | {:error, {atom(), map()}}
stat(path) :: {:ok, %{mtime: DateTime.t(), size: non_neg_integer()}} | {:error, {atom(), map()}}

# Shell behaviour:

```

### Tool Return Contract

All tools return `{:ok, result_map}` or `{:error, reason}`. The `result_map` should contain meaningful fields for the LLM to use in its next reasoning step.

### Command Parsing for `run_command`

The safety extension validates the full command string. The tool needs to parse it for `System.cmd`:
```elixir
# "mix test test/my_test.exs --trace" â†’ {"mix", ["test", "test/my_test.exs", "--trace"]}
[executable | args] = String.split(command)
shell().cmd(executable, args, [])
```

### `search_files` Strategy

Simple approach: list files, read each, grep for pattern. Not a full ripgrep replacement â€” just enough for agents to find code patterns. For MVP, non-recursive single-directory search. Can enhance later.

```elixir
def search_files(args, _context) do
  path = get_arg(args, :path)
  pattern = get_arg(args, :pattern)

  with {:ok, files} <- file_system().ls(path) do
    matches = files
    |> Enum.flat_map(fn file ->
      full_path = Path.join(path, file)
      case file_system().read(full_path) do
        {:ok, content} -> find_matches(full_path, content, pattern)
        _ -> []
      end
    end)
    {:ok, %{matches: matches}}
  end
end
```
