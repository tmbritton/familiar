# Story 9.5-3: Extension and MCP Tool Schema Registration

Status: done

## Story

As a Familiar extension author,
I want extensions and MCP servers to provide tool schemas alongside their tool registrations,
So that the LLM gets accurate parameter definitions for all available tools.

## Acceptance Criteria

1. **AC1: Extension tools with schemas.** `Extension.tools/0` accepts an optional 4th element: `{name, fun, description, parameters}`. The existing 3-tuple form still works. `ExtensionLoader` passes schemas through to `ToolSchemas`.

2. **AC2: ToolSchemas.register/2.** New public function `ToolSchemas.register(tool_name, schema)` merges a schema into the `:persistent_term` map at runtime. Used by both extension loading and MCP tool discovery.

3. **AC3: MCP tool schemas flow through.** When `MCP.Client.register_discovered_tools/2` processes a `tools/list` response, it extracts `inputSchema` from each tool and registers it via `ToolSchemas.register/2`.

4. **AC4: Precedence chain.** User `.familiar/tools/` file > MCP server-provided > extension-provided > compiled default. A user file override is never replaced by extension or MCP registration.

5. **AC5: knowledge_store schemas.** The `search_context` and `store_context` tools from KnowledgeStore extension have schemas registered and visible in `ToolSchemas.for_tools/1`.

6. **AC6: Extension type updated.** `Extension.tool_registration` type includes the 4-tuple form.

7. **AC7: Tests â€” extension schema registration.** Test that an extension providing a 4-tuple tool has its schema appear in `ToolSchemas.for_tools/1`.

8. **AC8: Tests â€” MCP schema registration.** Test that MCP-discovered tools have their `inputSchema` registered in `ToolSchemas`.

9. **AC9: Tests â€” precedence.** Test that user file overrides MCP and extension schemas; MCP overrides extension; extension overrides default.

10. **AC10: Tests â€” backward compatibility.** Existing 3-tuple extensions still work (no schema registered, falls back to default or empty).

11. **AC11: Clean toolchain.** `mix compile --warnings-as-errors`, `mix format --check-formatted`, `mix credo --strict`, `mix test`, `mix dialyzer` all pass.

12. **AC12: Stress-tested.** New tests pass 50 consecutive runs with no flakes.

## Dev Notes

### Current schema storage shape

`:persistent_term` stores `%{"tool_name" => %{description: ..., parameters: ...}}`. To support precedence, extend to `%{"tool_name" => %{description: ..., parameters: ..., source: :default | :file | :extension | :mcp}}`.

The `source` field is metadata for precedence decisions â€” `build_openai_schema/2` ignores it (only reads `description` and `parameters`).

### Precedence implementation

`register/3` checks the current source before overwriting:

```elixir
@source_priority %{default: 0, extension: 1, mcp: 2, file: 3}

def register(tool_name, schema, source) do
  schemas = :persistent_term.get(@persistent_key, %{})
  existing_source = get_in(schemas, [tool_name, :source]) || :default

  if @source_priority[source] > @source_priority[existing_source] do
    updated = Map.put(schemas, tool_name, Map.put(schema, :source, source))
    :persistent_term.put(@persistent_key, updated)
  end

  :ok
end
```

Note: `:file` sources are set during `load/1` and never overridden by `register/3` since file has the highest priority.

### Current extension tool registration flow

1. `ExtensionLoader.load_one/2` calls `mod.tools()` â†’ returns `[{:search_context, &fn, "desc"}]`
2. Maps to 4-tuple: `{:search_context, &fn, "desc", "knowledge-store"}`
3. `Application.register_extension_tools/1` calls `ToolRegistry.register/4` for each
4. No schema is passed â€” ToolSchemas has defaults from TOML files

After this story:
1. `mod.tools()` can return `{:search_context, &fn, "desc", %{"type" => "object", ...}}`
2. ExtensionLoader detects 4-tuple, calls `ToolSchemas.register("search_context", schema, :extension)`
3. Schema appears in `for_tools(["search_context"])`

### MCP tool discovery (client.ex lines 339â€“357)

`register_discovered_tools/2` iterates MCP `tools/list` response. Each tool has:
- `tool["name"]` â€” MCP tool name
- `tool["description"]` â€” human-readable
- `tool["inputSchema"]` â€” OpenAI-format JSON Schema (optional per MCP spec)

Tool is registered as atom `:"server_name__tool_name"` in ToolRegistry. The string form for ToolSchemas would be `"server_name__tool_name"`.
